# HTTP

> The agent-runtime server spine: a lean, production-shaped HTTP server over Node's `node:http`. A **`Server`** owns the lifecycle (listen on an ephemeral port → serve → graceful drain → close), a **`RouteManager`** resolves a request to a route + params (auto HEAD-from-GET, OPTIONS `Allow`, 404 / 405, groups), a Koa-onion **`MiddlewareManager`** wraps the handler, and a single unified **`RouteHandlerContext`** is the one surface every handler + middleware receives — parsed `method` / `url` / `query`, a lazy + cached `body()`, a mutable `state` bag for threading app deps, and `json` / `text` / `empty` / `error` responders. That is the whole core; the batteries are ten middleware factories (`createBodyParser` / `createErrorBoundary` / `createCors` / `createCompression` / `createTokenGuard` / `createRateLimiter` / `createSession` / `createCookieSession` / `createCSRFGuard` / `createStatic`) and `HTTPError` / `ContentTooLargeError` for client-facing faults. It also ships **content negotiation** over the weighted `Accept` family — a reusable **`Negotiator`** (`createNegotiator`) that picks the client's preferred media type / coding / language and `format`-dispatches to a handler map (or `406`) — and **transparent request decompression** folded into the body pipeline (a `Content-Encoding: gzip/deflate/br` request body is decoded before the handler ever sees it, with a zip-bomb guard on the decompressed size).
>
> It composes the core L1 primitives and adds **no new concurrency primitive**: it owns an [`Abort`](aborts.md) as the stop signal, arms a [`Timeout`](timeouts.md) for the graceful-drain grace, and exposes an [`Emitter`](emitters.md) for observability — the drain itself is a wake-park over a finish counter (`RequestManager`), never a busy-loop or a timer poll. Two invariants make it safe by construction: a handler throw is always caught into a `500` so it can never crash the process, and the `Server` is **observable** (AGENTS §13) — every event fires _after_ the relevant transition and a throwing listener is isolated, so a buggy observer can never reorder or corrupt the request / lifecycle engine.
>
> It is the **S2 middleware host** — the layer a browser app's requests land on before reaching the agent runtime — so it ships the gates a browser-facing edge needs: a stateless **token gate** (`createTokenGuard` over the `signToken` / `verifyToken` HMAC primitives — secret rotation, an HMAC-bound expiry, a total verify; [Tokens](#tokens)), per-client **rate limiting** (`createRateLimiter`), CORS, generic **sessions** (`createSession` over a pluggable `SessionStoreInterface` + `SessionTransport`, with both an in-memory and a durable DB-backed store), a signed-**cookie session** tier (`createCookieSession` — the same machine over a cookie transport, with `regenerate` / `destroy`), and **CSRF** (`createCSRFGuard` — stateless double-submit signed cookie). It also exposes two generic, protocol-agnostic seams the spine itself stays out of: the **Server-Sent-Events** streaming seam (`openSSEStream` / `bindStreamingAbort` — agent-agnostic; [Streaming (SSE)](#streaming-sse)) and the connection-handoff **upgrade** seam (`upgrade(handler)` fans a raw `node:http` upgrade out to registered `UpgradeHandler`s, first claimer wins; [Upgrade seam](#the-upgrade-seam)).
>
> The de-bloat is the design. The spine speaks NO WebSocket protocol — a [WebSocket transport](websocket.md) attaches THROUGH the upgrade seam rather than the spine learning RFC 6455 — and NO agent protocol — the SSE seam speaks only the wire, so a live agent turn and the future MCP HTTP transport are both _consumer_ patterns (the agent-stream route, the S2 proxy route) that live in the app layer, never `src/server/http`. No router DSL, no built-in auth coupling, no per-route framework magic: what ships is the lifecycle, the matcher, the onion, the one context, and the edge gates — nothing speculative. Source: [`src/server/http`](../../src/server/http). Surfaced through the `@src/server` barrel.

## Surface

Create a server, add middleware (outer-first) and routes, then `start` it on an ephemeral port:

```ts
import { createBodyParser, createCors, createErrorBoundary, createServer } from '@src/server'

const server = createServer()
server.use(createErrorBoundary()) // outermost — catches downstream throws
server.use(createCors())
server.use(createBodyParser())
server.route({
	method: 'POST',
	path: '/echo',
	handler: async (context) => context.json(await context.body()),
})

const port = await server.start() // ephemeral port (0 → bound port)
// fetch(`http://127.0.0.1:${port}/echo`, { method: 'POST', body: '{"hi":1}' })
await server.stop() // graceful drain, then close
```

A `Server` composes its `RouteManager` + `MiddlewareManager` internally and exposes a clean facade: `use` (a middleware), `upgrade` (a protocol-upgrade handler), `route` (one route or a batch), `group` (a prefixed registrar), and the §10 lifecycle verbs `start` / `stop` / `destroy`. Its `emitter` (the push observation surface, see [Observing](#observing)), `status`, and `port` members report live state. Each handler + middleware receives a `RouteHandlerContext` — the single unified request surface. By default `start` binds an ephemeral port on node's default interface; pass `createServer({ host, port })` to bind a fixed interface + port (a `0` / omitted port stays ephemeral), or call `discoverPort()` to pre-pick a guaranteed-free port.

### Factories

| API                          | Kind     | Summary                                                                                                                                                                                                                                                                                                    |
| ---------------------------- | -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `createServer`               | function | Create a `ServerInterface` — an observable lifecycle over `node:http`, composing route + chain.                                                                                                                                                                                                            |
| `createBodyParser`           | function | The JSON body-parsing `Middleware` — parses + size-limits the body (413 over the limit; 400 on malformed JSON).                                                                                                                                                                                            |
| `createErrorBoundary`        | function | The outermost error-catching `Middleware` — an `HTTPError` → its status, any throw → 500.                                                                                                                                                                                                                  |
| `createCors`                 | function | The CORS `Middleware` — sets the allow-origin header and answers an OPTIONS preflight.                                                                                                                                                                                                                     |
| `createCompression`          | function | The response-compression `Middleware` — buffers a non-streaming response, negotiates `gzip`/`deflate`/`br` from `Accept-Encoding`, compresses a worthwhile + compressible body (else passes through); bypasses streams.                                                                                    |
| `createNegotiator`           | function | Create a `NegotiatorInterface` — the reusable content-negotiation machine over the weighted `Accept` / `Accept-Encoding` / `Accept-Language` family.                                                                                                                                                       |
| `createTokenGuard`           | function | The S2 stateless-token `Middleware` — verifies the request's signed bearer or throws a `401`; stashes the value on `state`.                                                                                                                                                                                |
| `createRateLimiter`          | function | The per-client fixed-window `Middleware` — allows `max` requests per `window`, else a `429` + `Retry-After`.                                                                                                                                                                                               |
| `createSession`              | function | The generic session `Middleware` — resolve / mint / validate / `DELETE`-end + attach the session to `state`; generic over the session type, pluggable `store?` + `transport?`.                                                                                                                             |
| `createCookieSession`        | function | The cookie-transport session `Middleware` — `createSession` with the id in an HMAC-signed cookie (the same store / lifecycle, `regenerate` / `destroy`, way-out data persist).                                                                                                                             |
| `createCSRFGuard`            | function | The double-submit signed-cookie CSRF `Middleware` — a safe method mints + exposes a token, a mutating one verifies it against the signed cookie (else `403`); no server store.                                                                                                                             |
| `createStatic`               | function | The static-file `Middleware` (server-only `node:fs`) — serve a directory with a traversal guard, MIME, weak ETag/304, `Range`, dotfile policy, dir index, optional SPA fallback; composable (`next()` on no match).                                                                                        |
| `createMultipart`            | function | The streaming `multipart/form-data` `Middleware` (server-only `node:fs`/`os`/`crypto`) — stream-parses into `{ files, fields }`, stages each file under a RANDOM temp name, enforces DoS limits mid-stream + cleans temp files on breach/error/disconnect, magic-byte validates; non-multipart → `next()`. |
| `createSecurity`             | function | The security-header `Middleware` — sets a configurable secure-default set (`nosniff` / `X-Frame-Options` / opt-in HSTS / hardened CSP / `Referrer-Policy` / `Permissions-Policy` / COOP / CORP, opt-in COEP) + a sanitized `X-Request-ID` on `state`, then `next()`s; each header opt-out.                 |
| `createETag`                 | function | The conditional-GET `Middleware` for DYNAMIC output — REUSES `bufferResponse` to hash a GET 200 body into a (weak by default) `ETag` and answer `304` on a matching `If-None-Match`; passes non-GET/non-200/streaming through; composes INNER of `createCompression`.                                      |
| `createMemorySessionStore`   | function | Create the in-memory `SessionStoreInterface` (`createSession`'s default store) — a process-local `Map` + lazy idle-TTL; shareable across middlewares.                                                                                                                                                      |
| `createUploadedFile`         | function | Create one staged-upload `UploadedFileInterface` — a FROZEN, immutable data record (the sibling of core's `createFile`); `createMultipart` builds one per staged file part, its operations are the free functions below.                                                                                   |
| `createDatabaseSessionStore` | function | Create the driver-pluggable `SessionStoreInterface` over a `databases` table (driver default memory) — the durable twin replicating the idle-TTL + touch over a `lastSeen` column.                                                                                                                         |
| `openSSEStream`              | function | Open a Server-Sent-Events stream over a route's response — the generic, agent-agnostic streaming seam.                                                                                                                                                                                                     |
| `bindStreamingAbort`         | function | Bind a client disconnect to a once-only cancel; returns a leak-free cleanup (map disconnect → `agent.abort()`).                                                                                                                                                                                            |

## Methods

The public methods of each behavioral interface — every call-signature member listed (their `readonly` data members stay Surface rows). Each implementing class exposes exactly its interface's methods (AGENTS §22), except `RouteHandlerContext`, which adds internal spine-wiring methods (`assign` the matched params, `prepare` / `seed` the body) the public interface deliberately omits. `SSEStreamInterface`, `SessionTransport`, `SessionControlInterface`, and `ResponseBufferInterface` have no implementing class — each is an object literal a factory / the middleware returns — so they are method-documented but not held to a class match. `NegotiatorInterface` IS class-matched (`Negotiator` implements exactly it). `UploadedFileInterface` is a DATA-ONLY record (no methods — like `MultipartBody` / `MultipartPart`), so it is absent here; its operations are the free functions `streamUploadedFile` / `readUploadedFile` / `moveUploadedFile` (Surface rows).

#### `ServerInterface`

`use` / `upgrade` / `route` / `group` configure the server before `start`; `start` / `stop` / `destroy` are the §10 lifecycle verbs. The `id` / `status` / `emitter` / `port` members are Surface rows.

| Method    | Returns               | Behavior                                                                                                                                 |
| --------- | --------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `use`     | `void`                | Append a `Middleware` to the onion (outer-first — register the error boundary first).                                                    |
| `upgrade` | `void`                | Register an `UpgradeHandler` for raw protocol upgrades; the first to claim a socket owns it (see below).                                 |
| `route`   | `void`                | Register one `RouteInput` or a batch; a later route with the same method + path replaces the earlier.                                    |
| `group`   | `RouteGroupInterface` | Open a prefixed registrar — every route added through it (and nested groups) gets the prefix prepended.                                  |
| `start`   | `Promise<number>`     | Listen on the configured `host` / `port` (omitted / `0` port ⇒ ephemeral) and resolve the bound port; `idle` → `starting` → `listening`. |
| `stop`    | `Promise<void>`       | Stop accepting new requests, drain in-flight up to the `drain` deadline, then close; → `stopping` → `stopped`.                           |
| `destroy` | `Promise<void>`       | Final teardown — abort, force-close every socket, destroy the emitter; idempotent.                                                       |

#### `RouteManagerInterface`

`add` registers; `match` resolves a request; `allow` lists a path's methods (for OPTIONS); `group` opens a prefixed registrar. The `count` member is a Surface row.

| Method   | Returns                     | Behavior                                                                                                                                                                                                                                                   |
| -------- | --------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `add`    | `void`                      | Register one route or a batch; last write wins on a duplicate method + path.                                                                                                                                                                               |
| `match`  | `RouteMatch`                | Resolve a method + pathname to `matched` (route + params), `unmethoded` (405 + `allow`), or `unmatched` (404). The MOST SPECIFIC matching route wins (literal-over-param, order-independent); a trailing slash is insensitive; a HEAD matches a GET route. |
| `allow`  | `readonly HTTPMethod[]`     | The methods a pathname answers (including an implicit HEAD for a GET) — for an OPTIONS `Allow`.                                                                                                                                                            |
| `routes` | `readonly RouteInterface[]` | A snapshot of the registered routes in registration order.                                                                                                                                                                                                 |
| `group`  | `RouteGroupInterface`       | Open a prefixed registrar over this manager.                                                                                                                                                                                                               |
| `clear`  | `void`                      | Drop every registered route.                                                                                                                                                                                                                               |

#### `MiddlewareManagerInterface`

| Method    | Returns                      | Behavior                                                                                       |
| --------- | ---------------------------- | ---------------------------------------------------------------------------------------------- |
| `use`     | `void`                       | Append a `Middleware` (ordered — first registered is outermost).                               |
| `compose` | `(context) => Promise<void>` | Compose the chain into one runner ending in the given handler; a double-`next()` call rejects. |
| `clear`   | `void`                       | Drop every registered middleware.                                                              |

#### `RouteHandlerContextInterface`

The responders + `body()` (its `request` / `response` / `method` / `url` / `params` / `query` / `state` are Surface rows). `body()` collects + JSON-parses the request body once, cached; the responders write a complete response and are no-ops once the head is sent.

| Method  | Returns            | Behavior                                                                                                  |
| ------- | ------------------ | --------------------------------------------------------------------------------------------------------- |
| `body`  | `Promise<unknown>` | Lazily collect + decode the request body ONCE (JSON for a JSON content type, raw text otherwise), cached. |
| `json`  | `void`             | Send a JSON response with the given status (default `200`).                                               |
| `text`  | `void`             | Send a plain-text response with the given status (default `200`).                                         |
| `empty` | `void`             | Send an empty response (default `204`).                                                                   |
| `error` | `void`             | Send a JSON `{ error }` response with the given status.                                                   |

#### `SSEStreamInterface`

The write handle [`openSSEStream`](#streaming-sse) returns (its `closed` member is a Surface row). `write` serializes one `SSEMessage` to the wire (the `\n` → multiple `data:` lines split); `comment` writes a keep-alive the parser ignores; `end` ends the response. Every method is a no-op once `closed`.

| Method    | Returns | Behavior                                                                                                 |
| --------- | ------- | -------------------------------------------------------------------------------------------------------- |
| `write`   | `void`  | Serialize + flush one `SSEMessage` (its `data` split on `\n` into `data:` lines); a no-op once `closed`. |
| `comment` | `void`  | Write a `: text` SSE comment (a keep-alive the `SSEParser` ignores — no event); a no-op once `closed`.   |
| `end`     | `void`  | End the response, completing the stream; a no-op once already `closed`.                                  |

#### `SessionStoreInterface`

The pluggable storage seam behind [`createSession`](#) — the smallest set the middleware needs (AGENTS §21), the lean session analogue of [`QueueStoreInterface`](workers.md): just `get` / `set` / `delete` (no `load` / `clear` — the middleware never enumerates or bulk-clears). The STORE owns the idle-TTL policy (the expiry sweep + the touch-on-access live in `get`), so the session value stays pure identity. Every primitive is async, so a durable / DB-backed store fits the same shape as the default in-memory one. TWO stores implement it — the in-memory `MemorySessionStore` (the default) and the driver-pluggable `DatabaseSessionStore` (the durable twin), which REPLICATES the same idle-TTL + touch-on-access over a `databases` table (a `lastSeen` column). `get` / `set` carry an optional trailing `now` (epoch ms, default `Date.now()`) — the deterministic clock seam `createSession` feeds from its `clock` option, so an injected manual clock drives TTL eviction explicitly (no wall-clock race in a test).

| Method   | Returns                   | Behavior                                                                                                                                                      |
| -------- | ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `get`    | `Promise<S \| undefined>` | Resolve a LIVE session by id — applies idle-TTL expiry + touches `lastSeen` internally (against the optional trailing `now`); absent / expired → `undefined`. |
| `set`    | `Promise<void>`           | Insert / replace a session under its own `session.id` (no separate id param), stamping `lastSeen` with the optional trailing `now` (default `Date.now()`).    |
| `delete` | `Promise<void>`           | Drop a session by id (logout); an absent id is a no-op (no throw).                                                                                            |

#### `SessionTransport`

The id transport seam behind [`createSession`](#) — how the session id is READ from the request, WRITTEN to the response on a mint / rotation, and CLEARED on a `destroy`. It factors the wire location out of the session lifecycle so the SAME machine drives a HEADER session (the default, `headerSessionTransport`) or a signed-COOKIE one (`cookieSessionTransport`, for [`createCookieSession`](#)). Like `SSEStreamInterface` it has no implementing CLASS — `headerSessionTransport` / `cookieSessionTransport` return the object literal — so it is method-documented but not held to a class match. `read` is TOTAL (narrows untrusted input, never `as` — §14).

| Method  | Returns               | Behavior                                                                                                    |
| ------- | --------------------- | ----------------------------------------------------------------------------------------------------------- |
| `read`  | `string \| undefined` | Read the id off the request — TOTAL; an absent / malformed / tampered id → `undefined`.                     |
| `write` | `void`                | Write the id to the response (the mint / rotation path — a header set, or a signed `Set-Cookie`).           |
| `clear` | `void`                | Clear the id from the response (the `destroy` path — a `Max-Age=0` cookie; a no-op for a header transport). |

#### `SessionControlInterface`

The per-request lifecycle handle [`createSession`](#) stashes on `context.state` (under `SESSION_CONTROL_STATE`) beside the resolved session — the two niceties a handler invokes, which the middleware ACTS ON (the transport write runs synchronously at call time, the store I/O on the way out), keeping the `Session` entity store-decoupled (AGENTS §21). It is the object literal the middleware closes over — no implementing class, so method-documented but not held to a class match. Reached via `context.state.get(SESSION_CONTROL_STATE)`, narrowed with [`isSessionControl`](#). `destroy` supersedes `regenerate`.

| Method       | Returns | Behavior                                                                                               |
| ------------ | ------- | ------------------------------------------------------------------------------------------------------ |
| `regenerate` | `void`  | Flag the session for id ROTATION (anti-fixation) — mint a new id, carry the data across, drop the old. |
| `destroy`    | `void`  | Flag the session to END (logout) — `store.delete` + transport clear (a `Max-Age=0` cookie).            |

#### `NegotiatorInterface`

The content-negotiation machine [`createNegotiator`](#) returns (implemented by `Negotiator`, which exposes exactly these methods). `negotiate` is the generic media-type primitive (exact / subtype-wildcard / any-range, q-sorted, `;q=0` rejected); `encoding` / `language` are its sibling axes (a coding `*`, a language prefix match `en` ⇒ `en-US`) and REUSE the same q-value parser `createCompression` uses; `format` reads the request `Accept`, negotiates a `FormatHandlerMap`, and invokes the winner — or sends `406`.

| Method      | Returns               | Behavior                                                                                                         |
| ----------- | --------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `negotiate` | `string \| undefined` | Pick the best `available` media type for a weighted `Accept` header (exact / `type/*` / any-range), else absent. |
| `encoding`  | `string \| undefined` | Pick the best `available` content-coding for an `Accept-Encoding` header (a `*` wildcard ⇒ the first offered).   |
| `language`  | `string \| undefined` | Pick the best `available` language for an `Accept-Language` header (a primary tag `en` accepts `en-US`).         |
| `format`    | `Promise<void>`       | Dispatch to the handler whose media type the client most prefers from a `FormatHandlerMap`; `406` when none is.  |

#### `ResponseBufferInterface`

The reusable response-buffering handle [`bufferResponse`](#) installs behind [`createCompression`](#) (and a future ETag middleware) — it has NO implementing class (it is the object literal the helper returns), so it is method-documented but not held to a class match. `streaming` / `ended` / `status` are Surface rows. It captures a complete (non-streaming) response so the consumer can transform it before `send`, auto-bypassing to pass-through the instant the handler streams.

| Method    | Returns                              | Behavior                                                                                                                                                                            |
| --------- | ------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `headers` | `Record<string, string \| string[]>` | The captured response headers (lower-cased keys), merging `setHeader` + `writeHead`; a multi-valued header (notably `set-cookie`) stays a `string[]` so two cookies survive intact. |
| `body`    | `Buffer`                             | The concatenated buffered response body (empty for a body-less response).                                                                                                           |
| `send`    | `void`                               | Restore the real methods and write the final response — `writeHead(status? ?? captured, headers)` then `end(body)`; a `set-cookie` array rides through uncomma-folded.              |
| `release` | `void`                               | Restore the real methods + replay any captured head/chunks WITHOUT taking over — for an incomplete buffered response.                                                               |

## Contract

These invariants hold across `src/server/http` ↔ `http.md`:

1. **DOC ↔ SOURCE bijection.** Every `function` / `class` / `interface` / `type` row in the `## Surface` tables is a real export of the http module, and every export appears as a Surface row — exhaustive, both directions (AGENTS §22).
2. **Composes the core L1 primitives — no new concurrency primitive.** The `Server` owns a core [`Abort`](aborts.md) (the stop signal handed to in-flight handlers), arms a core [`Timeout`](timeouts.md) for the graceful-drain grace, and an [`Emitter`](emitters.md) for observability. The graceful drain itself is a wake-park over a finish counter in the `RequestManager` — recreated for request tracking, NOT a re-used `Queue`, but the same event-driven discipline. There is no busy-poll, no timer loop.
3. **The single unified `RouteHandlerContext`.** Every middleware and the matched route handler receive the SAME context instance per request — parsed `method` / `url` / `query` computed once, `params` filled by the matcher, a mutable `state` Map for threading app deps, and the responders. There is no separate per-middleware context shape.
4. **`body()` collects ONCE.** The first `body()` call collects + decodes the request body and caches the promise; every later call (a body-parser middleware then the handler) resolves the same cached value, so the consumable request stream is read exactly once — no double-read. The body-parser middleware drives this same path (applying its `limit`), it does not collect separately.
5. **The Koa onion + double-`next()` guard.** `MiddlewareManager.compose(handler)` threads the context through `middleware[0] → middleware[1] → … → handler`; each middleware runs code before and after `await next()`. A middleware that calls its `next()` more than once causes the runner to reject (`'next() called multiple times'`), so a link runs at most once and the ordering can't be corrupted. Omitting `next()` short-circuits the chain.
6. **Auto HEAD-from-GET + OPTIONS + 404 / 405.** A `HEAD` request matches a registered `GET` route (the handler runs; `node:http` discards the body). An `OPTIONS` request to a known path is answered `204` with an `Allow` header (the path's methods plus `OPTIONS`). A request whose pathname matches some route but not for its method is a `405` carrying the `Allow` set; a pathname matching no route is a `404`.
   6a. **Route precedence is literal-over-param SPECIFICITY, registration-order-INDEPENDENT.** The matching itself is DELEGATED to the shared [`@src/core` router engine](router.md) (the `RouteManager` holds a `RouteMatcher` keyed on its `Route` as the engine's `meta`): when several routes match the same concrete pathname, the engine returns the MOST SPECIFIC — it compares the candidates' per-segment specificity vectors (a LITERAL segment ranks above a PARAM segment; the comparator walks them left-to-right and the higher tier at the first differing segment wins). So `/users/me` resolves to the LITERAL route even when `/users/:id` was registered first, and the outcome is the same whichever was registered first (the Express / path-to-regexp default; we do NOT shadow a literal by an earlier-registered param). Identical method + path can't coexist (the `RouteManager` dedups by method + canonical path BEFORE handing entries to the engine — last write wins).
7. **Path params are compiled + URL-decoded; a trailing slash is INSENSITIVE.** All of this lives in the [`@src/core` router engine](router.md) — the server holds NO copy of the matcher. A `:name` segment becomes a capture group (the engine's `compilePath`), regex metacharacters in the literal parts are escaped, the pattern is anchored (matches the whole pathname, NOT a prefix — `/api` does not match the deeper `/api/users`), and matched values are URL-decoded (a malformed escape stays literal rather than throwing; a `%2F` in the raw path decodes to a literal `/` in the param value — Express-aligned, matched on the raw path). A SINGLE trailing slash is optional (Express's `strict: false` default): the engine normalizes the pattern's own trailing slash off (its `canonicalPath`) and emits an optional `/?` before the `$` anchor, so a route `/users` matches both `/users` and `/users/`, and `/users/:id` matches `/users/me` and `/users/me/` (same route + params). This is NOT prefix matching — only ONE trailing slash is folded (`/users//` still misses), and the ROOT `/` (and the empty pattern `''`) keep an exact anchor so `/` matches only `/`.
8. **Route groups share the one registry.** `group(prefix)` returns a `RouteGroup` that prepends the prefix and forwards to the parent `RouteManager` — grouped routes land in the SAME route table; the group is sugar for the shared prefix, not a separate registry. Nested groups concatenate prefixes.
9. **Lifecycle (§10) + configurable host/port bind.** `start` binds the configured `ServerOptions` `host` + `port` and transitions `idle → starting → listening`, resolving the ACTUALLY-bound port (read from the bound address via a guard); starting from `listening` rejects. An omitted / `0` `port` ⇒ an EPHEMERAL OS-assigned port (the default — every shipped server stands on one) and an omitted `host` ⇒ node's default interface, so the default `createServer()` is unchanged; a set `port` / `host` binds there (a port already in use rejects `start` with the `node:http` `EADDRINUSE` — no silent ephemeral substitution). Use `discoverPort` to pre-pick a guaranteed-free port. A `PortManager` / `Port` allocation registry is DEFERRED (AGENTS §21 build-with-consumer): one `Server` binds ONE port, so an allocation registry has no consumer — the `host` / `port` bind config + `discoverPort` cover the real need. `stop` transitions to `stopping`, fires the stop abort, drains in-flight requests up to the `drain` deadline (event-driven), then closes → `stopped`. `destroy` is the final teardown (abort + force-close + emitter destroy), idempotent.
10. **Graceful drain — event-driven.** On `stop`, the `RequestManager` parks on the drain deadline's signal and resolves the instant the last in-flight request finishes OR the deadline expires — never a busy-loop. If the deadline fires with requests still in flight, the server force-closes the sockets (`closeAllConnections`); a clean drain closes gracefully (idle keep-alive sockets are dropped so `close` never hangs). New connections after `stop` are refused.
11. **A handler throw never crashes the server.** Every unhandled throw escaping the middleware onion is caught and becomes a `500` (the [`createErrorBoundary`](#) middleware is the real handler, but the `Server` keeps a last-resort guard so an escaping throw — even past the boundary — can't crash the process). The server keeps serving subsequent requests.
12. **`HTTPError` is the client-facing fault.** A thrown `HTTPError` carries the HTTP `status` to send; the error boundary renders it as that status with the error's `message` (always client-facing — the handler's deliberate signal). `ContentTooLargeError` (413) is the body-parser's over-limit fault. Any OTHER throw renders as a `500` whose message is hidden unless `expose`. The `isHTTPError` guard narrows a caught value.
13. **CORS for the browser.** `createCors` sets `Access-Control-Allow-Origin` (a single origin, `'*'`, or a whitelist that echoes a listed request `Origin`) and answers an OPTIONS preflight `204` with the advertised methods + headers — so a browser app can call the server. **Two hardenings on the allow-LIST path** (where the ACAO REFLECTS the request `Origin`): the response carries `Vary: Origin` (appended via [`mergeVary`](#) without duplication) so a shared cache keys on `Origin` and cannot serve one origin's allowed response to another (cache poisoning) — a `'*'` / single-origin policy is a constant and adds NO `Vary`; and the literal `Origin: null` is NEVER reflected ([`resolveOrigin`](#) drops it unconditionally, even if `'null'` is on the list), so a sandboxed iframe / `file://` / opaque origin can't be granted access.
14. **DOC ↔ SOURCE method bijection.** The `## Methods` tables list exactly the public methods of each documented interface — exhaustive, both directions — and each implementing class (`Server` / `RouteManager` / `MiddlewareManager` / `RouteGroup` / `Route`) exposes the same public methods as its interface, no more (AGENTS §22). Two exceptions: `RouteHandlerContext` adds the internal `assign` / `prepare` spine-wiring methods the public `RouteHandlerContextInterface` omits; and `SSEStreamInterface` has no implementing class (it is the object literal `openSSEStream` returns), so it is method-documented but not held to a class match.
15. **Observable + scoped.** The `Server` owns a typed `Emitter` (AGENTS §13) exposed as `readonly emitter` (`ServerEventMap` — `start` / `stop` / `request` / `upgrade` / `error` / `drain`) and accepts the reserved `on?` initial-listeners hook plus the `error?` listener-error handler. **Emitting is observation-only** — the emitter isolates a listener throw (routing it to the `error` handler, never a domain event — `error` above is a server-level fault, distinct) and every event sits strictly AFTER the relevant lifecycle / request transition, so a buggy observer can NEVER reorder, throw into, or corrupt the request / lifecycle engine. NO auth baked in.
16. **The SSE seam is generic + agent-agnostic.** `openSSEStream` / `bindStreamingAbort` (in `src/server/http`) speak only HTTP + the SSE wire (`SSEMessage`) — there is NO `@src/core/agents` import in the http module — so the seam is reusable (the future MCP HTTP transport). `openSSEStream` writes spec-correct SSE that is the EXACT inverse of the core [`SSEParser`](parsers.md): each `write` splits `data` on `\n` into one `data:` line per segment (round-tripping through the parser's multi-`data` concat), with `event:` / `id:` / `retry:` lines for the present fields and a terminating blank line; `comment` is a parser-ignored keep-alive; every method is a no-op once `closed`. `bindStreamingAbort` fires its `onAbort` exactly once on the first client-disconnect signal and returns a leak-free cleanup. An open SSE stream interacts correctly with the graceful drain: a consumer's `stop`-event wiring ends it promptly, and the spine's deadline force-close is the backstop, so `server.stop()` always resolves and never hangs (see [Streaming (SSE)](#streaming-sse)). The agent-stream route that maps `AgentChunk` → SSE is a CONSUMER pattern (the guide + the live test), NOT a framework export.
17. **Stateless signed-token validation (the S2 gate).** A token is `<base64url-payload>.<base64url-signature>` where the payload is a JSON `{ value, exp? }` and the signature is its HMAC-SHA256 under a secret. `signToken` signs with the CURRENT secret (a single string, or the first of a `[current, ...older]` rotation list); `verifyToken` (and `createTokenGuard`) accept a token signed by ANY secret in the list, so a key rotates with zero downtime (prepend the new secret, drop the old once outstanding tokens expire). When a `ttl` is set the expiry instant is bound INSIDE the signed payload, so it is HMAC-covered and cannot be lengthened by tampering; an elapsed expiry rejects. `verifyToken` is TOTAL — a malformed token, a bad signature (compared length-guarded + constant-time via `timingSafeEqual`), a hostile / non-JSON payload, an empty secret list, or an expired token all yield `undefined`, never a throw (AGENTS §14); only `signToken` throws, and only on a misconfigured empty rotation list (a `500`, programmer error). `createTokenGuard` reads the token from `header` (default `authorization`, lower-cased), strips the `scheme` prefix (default `Bearer`, case-insensitive; `''` ⇒ the raw value is the token), and on a valid token stashes the verified value on `context.state` under `TOKEN_STATE` then continues the chain; a missing token throws `HTTPError(401, 'missing token')` and an invalid/expired/tampered one `HTTPError(401, 'invalid token')` — both rendered by [`createErrorBoundary`](#) (the failure path never writes the response directly). The S2 proxy route that validates a browser's obfuscated bearer then injects the real LLM key server-side is a CONSUMER pattern (the guide + the live test), NOT a framework export — exactly like the agent-stream route.
18. **Per-client fixed-window rate limiting.** `createRateLimiter({ max, window })` allows exactly `max` requests per `window` (ms) per client key, counting each key against ONE composed core [`Budget`](budgets.md) (charged one unit per request) paired with the window's reset instant. The check is BEFORE the consume (`budget.exhausted` is `consumed >= max`, read synchronously — never racing a signal), so the `max + 1`th request in a window is the first denied. The window rolls LAZILY — the first request after a key's window has elapsed `clear()`s that same key's Budget (the §10 reset — zeroes the tally + re-arms the signal, starting the next window from zero) rather than allocating a new one; a key's FIRST request ever creates its single Budget. There is NO background timer (a middleware factory has no teardown hook). ALL window math reads the injected `clock` (a `() => number`, default `Date.now` — `SessionOptions`' deterministic clock seam, §16): a test drives a window roll by advancing a manual clock explicitly, never racing a real window against wall-clock; production never sets it. The over-limit request short-circuits: it sets `Retry-After` (the whole seconds until reset, floored at 1) and writes a `429` with `message` (default `DEFAULT_RATE_LIMIT_MESSAGE`) WITHOUT calling `next()` — set-header-then-respond like [`createCors`](#), so the header is unambiguously preserved; it does NOT throw, so no error boundary is required. Memory is bounded by `capacity` (default `DEFAULT_RATE_LIMIT_CAPACITY`): inserting a brand-new key at the cap first evicts the oldest-inserted one. The default key is the verified token (when an upstream [`createTokenGuard`](#) stashed one under `TOKEN_STATE`) else the raw socket peer IP — narrowed with `typeof`, never `as` (§14), and NEVER a forwardable header (`X-Forwarded-For` is spoofable, the same proxy-SSRF caution the S2 proxy note carries); a deployment behind a trusted proxy supplies its own `key`.
19. **The upgrade seam is generic + protocol-agnostic.** `upgrade(handler)` registers an `UpgradeHandler` for a raw `node:http` protocol upgrade (a request with `Connection: Upgrade`). When an upgrade arrives the spine fans it out to the registered handlers in registration order, handing each the raw triple — the upgrade `request`, the upgraded `node:stream` Duplex `socket`, and the buffered `head` (node's own types, narrowed at the boundary, never asserted — §14). The FIRST handler that returns `true` CLAIMS the socket: it now OWNS the connection (it writes the `101` handshake and speaks its own wire protocol), no later handler runs, and the spine leaves the socket alone. A handler returning falsy DECLINES, so the next registered handler gets a chance. If NO handler claims it (none registered, or all declined) the spine `socket.destroy()`s the un-upgraded connection — an unhandled upgrade never leaks a dangling socket. An `upgrade` event then fires (strictly AFTER the fan-out resolves; the emitter isolates a listener throw) carrying the request + whether it was `handled`, for observability only — a buggy observer can neither claim the socket nor change the decision. The spine speaks NO WebSocket / upgrade protocol itself; a transport that does (a WebSocket transport built on [`createNodeWebSocket`](websocket.md)) attaches THROUGH this seam (see [The upgrade seam](#the-upgrade-seam)). The upgrade listener is bound per `start()` run and discarded with the closed server on `stop` / `destroy`, so it never leaks across a restart.
20. **Generic, reusable sessions.** `createSession({ header?, ttl?, key?, create?, mint?, require?, onMissing?, store?, clock? })` is the spine's session primitive — generic over the session type `S extends SessionInterface` (default `Session`). The per-id storage is a pluggable `SessionStoreInterface` (clause 21) — the middleware no longer owns the `Map` or the TTL math; it just calls `store.get` / `store.set` / `store.delete`. The store applies the lazy idle-TTL eviction: a session not accessed within `ttl` reads as ABSENT and is dropped on access (NO background timer), and every live access TOUCHES it so an active session never ages out. Per request `createSession` resolves the id from `header` (default `DEFAULT_SESSION_HEADER` `'session-id'`, the value `string | string[] | undefined` narrowed with `isString`, never `as` — §14) and: a VALID session (resolved via `store.get`) is set on `context.state` under `key` (default `SESSION_STATE`), and the chain proceeds; with no valid session, if `mint(context)` returns true a fresh session is MINTED (`crypto.randomUUID()` id, `create(id)` — default a `Session` — `store.set`, the id SET on the response `header` which `writeHead` merges, set on `context.state`, proceed); a `DELETE` carrying a valid id ENDS the session (`store.delete` → `204`, the generic logout / teardown — checked BEFORE the proceed / mint paths so a valid session is ended, not passed on); with no valid session and not minting, `require` rejects via `onMissing(context)` (default a plain `404`) while an unset `require` just proceeds sessionless (nothing set on `context.state`). The default `mint` ALWAYS mints (the AUTO-SESSION — a request with no valid id silently opens a new session); a consumer narrows it (`mint` may be async and read the CACHED `context.body()`, so a downstream route re-reads the same body). `SessionInterface` is minimal — a single `readonly id` (the store tracks `lastSeen` ITSELF, so the session value holds only identity) — and the default `Session` adds a `data` Map for per-session app state; a consumer reads the resolved session off `context.state`, narrowing with `instanceof` / a guard (never `as`). It is MECHANISM, not policy, and ADDITIVE (omit it and the server is sessionless). The MCP transport's [`createMCPSession`](mcp.md) is the canonical consumer — it CONFIGURES this with the `mcp-session-id` header, a mint-on-`initialize` predicate, `require`, and an `MCPSession` `create` (it takes the DEFAULT memory store via `ttl`), then adds ONLY the resumable `GET` SSE stream; the generic resolve / mint / validate / `DELETE`-end / TTL machinery is `createSession`'s, NOT MCP-bespoke.
21. **The session store is a pluggable seam.** `SessionStoreInterface<S>` is the smallest storage set `createSession` needs (AGENTS §21) — `get` (resolve a live session, the store applying idle-TTL expiry + the touch-on-access internally) / `set` (insert / replace under the session's OWN `id`, no separate id param) / `delete` (drop by id) — the LEAN session analogue of [`QueueStoreInterface`](workers.md) (no `load` / `clear`, since the middleware never enumerates or bulk-clears). Every primitive is async, so a durable backend (a DB round-trip) fits the same shape as the in-memory one. `createSession` defaults to `createMemorySessionStore({ ttl, absoluteTtl })` — `MemorySessionStore`, a process-local `Map<string, { session, lastSeen, createdAt }>` (the LIFTED closure store, the same lazy-window idiom as [`createRateLimiter`](#)); its `get` / `set` carry an optional trailing `now` (epoch ms, default `Date.now()`) — the deterministic clock seam ON the `SessionStoreInterface` signatures, which `createSession` feeds from its `clock` option (a `() => number`, default `Date.now` — threaded into EVERY store `get` / `set` the middleware makes), so a test injects a manual clock and drives TTL eviction by advancing it explicitly rather than racing a real timer against wall-clock (§16). **An optional `absoluteTtl` caps the TOTAL lifetime from mint, independent of the idle `ttl` touch**: a session older than `absoluteTtl` (`now - createdAt >= absoluteTtl`) is evicted on the next `get` EVEN IF continuously active, bounding a hijacked / long-lived id (the idle `ttl` alone would keep a touched session alive forever). `createdAt` is stamped ONCE at the first `set` (the mint) and PRESERVED across the way-out re-persists, so activity never slides the absolute window. The `DatabaseSessionStore` twin replicates BOTH cursors over its `lastSeen` + `createdAt` columns identically. A custom `store` REPLACES the default without touching `createSession` (the store OWNS the TTL, so `ttl` is consumed by the DEFAULT store and ignored once a `store` is supplied), and one `createMemorySessionStore()` can be SHARED across several session middlewares (proven in the tests). The seam ships TWO stores: the in-memory `MemorySessionStore` (the default) and the driver-pluggable `DatabaseSessionStore` (`createDatabaseSessionStore`, the durable twin — the session-store mirror of the `QueueStore` / `WorkflowStore` dual-store pair). `DatabaseSessionStore` REPLICATES the memory twin's exact idle-TTL contract over one `databases` table: the row is `{ id; session; lastSeen }` ([`SessionRow`](#)) — the session held as ONE OPAQUE JSON column (a `rawShape`, narrowed back on `get` by the floor guard [`isSession`](#), the §14 boundary narrow; the opaque column keeps the row type flat for the OPEN generic `S`, sidestepping a TS2589 a typed-`S` column would spring) and `lastSeen` a `number` column holding the same store-owned idle cursor — so `get` evicts an idle row (`now - lastSeen >= ttl` → `remove`, reads ABSENT), TOUCHES a live one (`update` `lastSeen = now`, so an active session never ages out), and with no `ttl` never expires, BYTE-FOR-BEHAVIOR identical to the memory twin (proven equivalent by the same injected-`now`-clock test). The `driver` DEFAULTS to memory so it works out of the box; a server SQLite / browser IndexedDB driver makes sessions durable across restarts — WITHOUT touching `createSession`. A persisted session must be a JSON-serializable `SessionInterface` (the same caveat the `QueueStore`'s `input` carries): in particular the default `Session` CLASS does NOT round-trip — its `id` lives in a `#private` field behind a getter that serialization strips — so a durable consumer stores a PLAIN session record. (The signed-COOKIE session tier + CSRF now ship on this store — see clauses 23 / 24 — built on net-new cookie machinery.)
22. **The id transport is a pluggable seam.** `SessionTransport` (`read` / `write` / `clear`) factors WHERE the session id rides out of `createSession`'s lifecycle, so the SAME resolve / mint / validate / TTL / `regenerate` / `destroy` / persist machine drives a HEADER session or a signed-COOKIE one. `createSession` defaults to `headerSessionTransport(header)` — the pre-existing behavior unchanged (read / set the `header`; `clear` is a no-op, a header carries no stored state to revoke), so every shipped header-session test stays green. A consumer passes a `transport` to swap the wire location; [`createCookieSession`](#) passes `cookieSessionTransport(name, secret, cookie)`. `read` is TOTAL — it narrows untrusted input (a raw header, a cookie string) with a guard, never `as` (§14) — so an absent / malformed / tampered id reads as no session. Beyond the transport, two way-out behaviors are now part of every `createSession`: the resolved session is PERSISTED (`store.set` after `next()`), so a DURABLE store round-trips a handler's `data` changes across requests (the in-memory store re-stamps the same object — harmless); and a `SessionControlInterface` (`regenerate` / `destroy`) rides `context.state` under `SESSION_CONTROL_STATE` for the handler to flag (narrowed with [`isSessionControl`](#), never `as`) — `regenerate()` ROTATES the id (mint a new id, carry the `data` across via `transferSessionData`, `store.set` under the new id, `store.delete` the old — anti-fixation), `destroy()` ENDS it (`store.delete` + transport clear). Each does its TRANSPORT write SYNCHRONOUSLY at call time (a `Set-Cookie` must land before the handler sends its response) and defers only the async store I/O to the way out; `destroy` supersedes `regenerate`. The MCP session middleware is unaffected — it uses the default header transport, never flags the control, and the way-out `store.set` re-stamps its in-memory `MCPSession` harmlessly.
23. **Cookie sessions — auto-secure + cookie-injection hardened.** `createCookieSession({ secret, name?, cookie?, ttl?, absoluteTtl?, key?, create?, mint?, require?, onMissing?, store?, clock? })` is `createSession` with the id carried in an HMAC-SIGNED cookie instead of a header — it CONFIGURES the generic machine with `cookieSessionTransport`, so every store / lifecycle behavior (clauses 20–22) is inherited, NOT re-implemented. A request's id rides the `Cookie` header (the cookie value is `signToken(id, { secret })`, so a signed cookie is just a stateless token in a cookie — same secret rotation + tamper rejection, no second HMAC scheme); the mint / rotation sets a signed `Set-Cookie` (name default `DEFAULT_SESSION_COOKIE` `'session'`, attributes from `cookie` — `path` default `/`, `HttpOnly` on, `SameSite=Lax`); a `destroy` / `DELETE` clears it (`Max-Age=0`). The cookie's `Max-Age` is derived from `ttl` (ms → whole seconds, so the cookie expires with the session) unless `cookie.maxAge` overrides it. A tampered / absent cookie reads as NO session (a fresh one is minted). **`secure` defaults to AUTO** ([`resolveSecure`](#) over [`requestEncrypted`](#)) — `Secure` when the request arrived over TLS (`request.socket.encrypted === true`), off over plaintext HTTP, so a direct-HTTPS deployment is `Secure` automatically while local dev still works; set `cookie.secure: true` to FORCE it behind a TLS-terminating proxy (where `'auto'` reads no TLS), or `false` to force it off. **The cookie machinery is injection-hardened** (shared with CSRF + every signed cookie, clause 21's helpers): [`serializeCookie`](#) VALIDATES `Domain` / `Path` (throws on a `;` / `,` / whitespace / control char that could inject another attribute) and FORCES `Secure` on a `sameSite: 'None'` cookie; [`parseCookies`](#) REJECTS a non-token / whitespace-padded NAME on the way in, so a `'  __Host-x'` can't be trimmed into the prefix-protected name. It composes like any middleware (an error boundary / CORS / a guard sit IN FRONT); a handler reads the session off `context.state` (narrowed with `instanceof` / a guard, never `as`) and may `regenerate()` / `destroy()` via the control handle. `flash` is DEFERRED — the `data` bag covers cross-request state; redirect-flash serves server-rendered PRG, not taverna's SPA + JSON-API consumer (built with its consumer, §21).
24. **Static-file serving — traversal-guarded + composable.** `createStatic({ root, prefix?, index?, dotfiles?, maxAge?, etag?, fallback? })` serves a directory of files over HTTP using server-only `node:fs`, COMPOSABLY: it serves `GET` / `HEAD` only (any other method falls through `next()`; a `HEAD` serves the headers with NO body), and a request that does NOT resolve to a served file falls through via `next()` (so other routes / middleware still run) — it is not terminal, unless it served, the SPA `fallback` claimed it, or the `dotfiles` policy denied it (`403`). The SECURITY-CRITICAL piece is the TRAVERSAL GUARD ([`resolveStaticPath`](#)): the request path is stripped of `prefix`, `decodeURIComponent`d (a malformed escape ⇒ not-found), NUL-byte-refused, made RELATIVE (leading separators stripped FIRST so a leading `..` survives normalization as a climbing segment rather than being silently swallowed against an absolute root), `normalize`d, then `resolve`d under the already-resolved `root` and REQUIRED to equal `root` or sit beneath `root + path.sep` — so a `../`, an encoded `..` (`%2e%2e`), an absolute decoded path, a NUL byte, or a sibling that merely shares the root's string prefix all resolve OUTSIDE and are refused (the file is never `stat`ed, never read). It is TOTAL — adversarial input never throws (AGENTS §14). Per request it: gates the method; resolves the path under `root`; applies the `dotfiles` policy to a dot-segment path ([`isDotfilePath`](#)) — `'ignore'` (default `DEFAULT_DOTFILES`) ⇒ fall through, `'deny'` ⇒ `403`, `'allow'` ⇒ serve; `stat`s the target (ENOENT / a non-file ⇒ the fallback / `next()`; a DIRECTORY ⇒ serve its `index`, default `DEFAULT_STATIC_INDEX` `'index.html'`, else fall through); and serves a file with its extension-derived `Content-Type` ([`lookupContentType`](#) over the frozen `EXTENSION_TYPES`, default `DEFAULT_CONTENT_TYPE`), an optional `Cache-Control: max-age=<maxAge>` (seconds; omitted ⇒ no header), a WEAK `ETag` ([`computeWeakETag`](#) `W/"<size>-<mtimeMs>"`, default on — a matching `If-None-Match` ⇒ `304` with no body; `etag: false` disables it), `Accept-Ranges: bytes`, and HTTP `Range` support ([`parseRange`](#) — a satisfiable range ⇒ `206` + `Content-Range: bytes start-end/size` + the sliced `fs.createReadStream(path, { start, end })`, an unsatisfiable one ⇒ `416` + `Content-Range` with the asterisk form, no / multi-range ⇒ the full `200`). The body STREAMS via `fs.createReadStream` piped to the raw response (`writeHead` first); a mid-stream read error destroys the response rather than crashing the process (stream-error-safe). When `fallback` is enabled (`true` ⇒ on with the default `/api` exclude `DEFAULT_STATIC_FALLBACK_EXCLUDE`, an object overrides `exclude`, omitted / `false` ⇒ off), a GET that did NOT resolve to a file, is extension-less, accepts `text/html`, and is not under `exclude` is served the `index` document (so a client-side router owns the route — the SPA shell); otherwise `next()`. Every untrusted input (the decoded path, the `Range` / `If-None-Match` / `Accept` headers) is narrowed with `typeof`, never `as` (§14). It is MECHANISM, not policy — it is a plain server middleware over `node:fs`, NOT the deferred Filesystem / Sandbox abstraction (there is no browser static-serving peer, so no cross-env interface). The headline use is co-serving a built SPA in front of a JSON / SSE API (the SPA fallback owns client routes; a `/api` route 404s honestly).
25. **CSRF — double-submit signed cookie, SESSION-BOUND.** `createCSRFGuard({ secret, cookie?, header?, field?, key?, session?, safe? })` is the stateless double-submit-cookie defense — NO server-side store (the signed cookie IS the state). On a SAFE method (`safe`, default `GET` / `HEAD` / `OPTIONS`) it MINTS a token, sets it as a SIGNED cookie (name default `DEFAULT_CSRF_COOKIE` `'csrf'`, `SameSite=Strict`, `HttpOnly:false` so client JS can READ it to echo, `Secure` AUTO — clause 23's per-request TLS resolution), and exposes the RAW token on `context.state` under `key` (default `CSRF_STATE` `'csrf'`) for the handler to embed; the chain proceeds. On a MUTATING method it reads the submitted token (from `header`, default `DEFAULT_CSRF_HEADER` `'x-csrf-token'`, else the parsed-body `field`, default `DEFAULT_CSRF_FIELD` `'_csrf'` — which needs [`createBodyParser`](#) ahead), [`readSignedCookie`](#)s the cookie's value, and [`safeCompare`](#)s the two; a missing / invalid (tampered) cookie, a missing submitted token, or a mismatch throws `HTTPError(403, 'invalid csrf token')` (rendered by [`createErrorBoundary`](#) — register it OUTSIDE), else `next()`. **SESSION-BINDING** closes the cookie-injection gap a bare double-submit leaves (a sibling-subdomain / MITM attacker who can WRITE a cookie can plant both halves): when a session is on `context.state` under `session` (default `SESSION_STATE`, the key the session middleware set — so register the guard AFTER `createSession` / `createCookieSession`), the minted token is BOUND to the session id (`signToken(sessionId)`) and a mutating request must present the token whose recovered bound id ([`verifyToken`](#)) equals ITS OWN session id — so a token minted under session A is `403`ed when replayed on a request authenticated as session B, even with matching double-submit halves. With NO session ahead it FALLS BACK to a signed-random token (the plain double-submit, weaker — no per-user binding, which is why the session ordering is required). It is MECHANISM (pair it with a session / auth layer). Every read narrows untrusted input with `typeof` / `isRecord` / `isSession`, never `as` (§14).
26. **Response compression — buffer, negotiate, compress; streams pass through.** `createCompression({ threshold?, encodings? })` BUFFERS a non-streaming response (via [`bufferResponse`](#), which wraps `writeHead` / `write` / `end` / `flushHeaders`), runs the chain, then negotiates the best [`Encoding`](#) from the request `Accept-Encoding` ([`negotiateEncoding`](#) over the SHARED [`parseAcceptHeader`](#) q-value parser — server `encodings` order breaks a client tie, `*` + `;q=0` honored) and, IF the buffered body is at or above `threshold` (default `DEFAULT_COMPRESSION_THRESHOLD` `1024`) AND its `Content-Type` is compressible ([`isCompressibleType`](#) — `text/*`, a `+json`/`+xml` suffix, or an explicit `COMPRESSIBLE_TYPES`; NOT images / video / already-zipped), COMPRESSES it via `node:zlib` ([`compressBody`](#)) and sets `Content-Encoding` + `Vary: Accept-Encoding` ([`mergeVary`](#)) + the new `Content-Length` (DROPPING the stale one). Otherwise the buffer is sent UNCHANGED — below threshold, an incompressible type, an already-`Content-Encoding`-set response, `Accept-Encoding: identity` / none, a `HEAD` request, or a `204` / `304` (no body) all pass through. **It does NOT break streaming / SSE.** An unbounded stream cannot be buffered, so the buffer AUTO-BYPASSES to pass-through the instant the handler streams — a `writeHead` whose `Content-Type` is `text/event-stream` (an [`openSSEStream`](#) response), an explicit `flushHeaders()`, or a raw `write()` before `end()` (a chunked producer; the senders never call `write()`); on bypass it restores the real methods and replays the captured head + chunks straight to the socket, so the stream proceeds untouched. Register it near the TOP of the chain (OUTER of the routes AND OUTER of [`createErrorBoundary`](#), so the boundary's error body — written inner — is compressible too); if compression sits INNER of the boundary, a downstream throw unwinds past it and the buffer is [`release`](#)d untouched (so the boundary writes the error raw). Every header is narrowed at the boundary, never `as` (§14). Proven over a real server: `fetch` keeps the `Content-Encoding` header + decodes the body, and a raw `node:http` read (`readRawResponse`) confirms the compressed wire size + `node:zlib` decode.
27. **Transparent request decompression — on the one `body()` path, with a zip-bomb guard.** A request body declaring `Content-Encoding: gzip` / `deflate` / `br` (narrowed by [`requestEncoding`](#) — a multi-coding / unknown / `identity` is left as-is) is DECOMPRESSED ([`decompressBody`](#)) BEFORE the content-type decode, so the handler's `context.body()` sees the decoded value transparently (clause 4's single cached collection path now collects → decompress → decode). This lives on the `RouteHandlerContext.body()` path itself, so it applies whether or not [`createBodyParser`](#) is mounted — the body parser only DRIVES that path eagerly with its `limit`. The size `limit` is enforced on the DECOMPRESSED size (the zip-bomb-safe default — a small compressed payload that inflates past `limit` is a `413`), IN ADDITION to `collectBody` bounding the compressed input bytes. A corrupt compressed body throws (a `gunzip` / `inflate` error), which the body parser maps to a `400` (a client fault), not a `500`. Proven with a real `node:zlib`-compressed request body and a real zip bomb (a sub-KiB payload inflating to ~100 KB → `413`).
28. **Content negotiation — one machine, the shared q-value parser.** [`createNegotiator()`](#) returns a `Negotiator` — a reusable, cross-middleware ENTITY (NOT a middleware) over the weighted `Accept` family. `negotiate(header, available)` picks the client's most-preferred media type the server offers (an exact match, a `type/*` subtype wildcard, or the any-media-range ⇒ the first offered; q-sorted; a `;q=0` entry never matches); `encoding` / `language` are the sibling axes (a coding `*` wildcard, a language primary-tag match `en` ⇒ `en-US`); `format(context, handlers)` reads the request `Accept` (an ABSENT header ⇒ the any-range ⇒ the first handler key), negotiates a [`FormatHandlerMap`](#)'s media types, and invokes the winning responder — or sends `406 Not Acceptable` via `context.error` when none is acceptable. It REUSES the SAME `parseAcceptHeader` q-value parser [`createCompression`](#) uses — one parser, never two — and `Negotiator` implements exactly `NegotiatorInterface` (no extra public surface). Every request header is narrowed (`typeof`), never `as` (§14).
29. **Streaming multipart uploads — staged, DoS-bounded, magic-byte-validated, leak-free.** [`createMultipart({ limits?, allowed?, directory? })`](#) STREAM-parses a `multipart/form-data` request body (server-only `node:fs` / `os` / `crypto`) — it does NOT go through the buffering `context.body()`; it reads the raw `request` through a boundary-scanning state machine ([`parseMultipartRequest`](#)) that NEVER buffers the whole body, then SEEDS the parsed `{ files, fields }` ([`MultipartBody`](#)) as `context.body()` (the handler reads `await context.body()`, narrowing with [`isMultipartBody`](#), never `as` — §14). The boundary is extracted by [`multipartBoundary`](#) from the raw `Content-Type`; a NON-multipart request passes through untouched (`next()`). Each part's headers ([`parsePartHeaders`](#)) route a FILE part (a `Content-Disposition` carrying a `filename`) to a staged file (a frozen [`UploadedFileInterface`](#) record built by [`createUploadedFile`](#)) and a FIELD part (no filename) to a collected string (a repeated field keeps the LAST). **DoS limits ([`MultipartLimits`](#)) are enforced MID-STREAM** — `file` / `files` / `field` / `fields` / `total` (defaults `DEFAULT_MULTIPART_*`), each tripping the moment it is exceeded: reading STOPS, every already-staged temp file is DELETED, and a [`MultipartError`](#) `413` is thrown (rendered by [`createErrorBoundary`](#)); nothing is buffered past the bound. **Magic-byte validation ([`detectMIME`](#) over the frozen [`MAGIC_BYTES`](#)) is the anti-spoofing defense** — each file's leading bytes are sniffed and recorded as `validated`; with an `allowed` whitelist, a file whose REAL (sniffed) type is not on the list OR whose declared `Content-Type` does not match its magic bytes is REJECTED (the parse aborts with a `MultipartError` `415` after cleaning every staged temp file — fail-closed); with NO whitelist the `mime` + `validated` are recorded but no file is rejected on type. **Each file is staged under a RANDOM temp name** (`crypto.randomUUID()` in `directory`, default `os.tmpdir()`) — the untrusted client `filename` is METADATA only (the `name` field) and is NEVER used to build the path, so a `filename="../../etc/x"` cannot escape the temp dir (the path-traversal defense). **Cleanup leaks no temp file**: on a limit breach, a parse error, OR a client disconnect mid-upload (bound to the request's `'aborted'` event — the precise mid-message signal, which a clean upload never trips), every staged temp file is deleted (the part's open write stream is destroyed first, so the unlink succeeds even on Windows). After a SUCCESSFUL parse the APP OWNS the staged files (frozen, immutable [`UploadedFileInterface`](#) records) — they persist until the handler [`moveUploadedFile`](#)s them (a `fs.rename`, with an `EXDEV` cross-device copy + unlink fallback; it returns a NEW frozen record with `status` → `'moved'`, the original unchanged) or deletes them; the handler reads the bytes via the free functions [`streamUploadedFile`](#) / [`readUploadedFile`](#). `UploadStatus` is `'staged'` (a clean parse) → `'moved'` (the moved record); a rejected / breaching upload never hands the handler a bad file. It is MECHANISM, not policy (a plain server middleware over `node:fs`, with NO virus scanning / storage backend baked in). A malformed body (a missing / unterminated boundary, a nameless part) is a `400`. The `MultipartError`'s `reason` ([`MultipartReason`](#)) names the fault (`limit` / `malformed` / `rejected`); narrow it with [`isMultipartError`](#). Proven over a real server with real `FormData` / raw multipart bodies (no mocks).
30. **Security headers — a configurable secure-default set + a correlation id.** [`createSecurity({ frame?, hsts?, csp?, referrer?, permissions?, requestId? })`](#) sets a conventional secure-default response-header set then `next()`s — the set-header-then-continue shape of [`createCors`](#), composing anywhere (register it near the TOP so every response carries the headers). `X-Content-Type-Options: nosniff` is UNCONDITIONAL (no value to tune, no downside). Every other header is opt-OUT: `frame` ⇒ `X-Frame-Options` (default `DEFAULT_FRAME_OPTIONS` `'DENY'`; `'SAMEORIGIN'`, or `false` to omit); `csp` ⇒ `Content-Security-Policy` (default `DEFAULT_CSP` `"default-src 'self'"`); `referrer` ⇒ `Referrer-Policy` (default `DEFAULT_REFERRER_POLICY`); `permissions` ⇒ `Permissions-Policy` (default `DEFAULT_PERMISSIONS_POLICY`) — each a `string` OVERRIDE, `false` to OMIT, or the default when unset (the [`resolveSecurityHeader`](#) value-or-off union, AGENTS §4.4, NOT a behavioral toggle). `hsts` ⇒ `Strict-Transport-Security` is OFF by default (it is HTTPS-only and browser-PINNED, so a wrong value is sticky): `true` sets `DEFAULT_HSTS` (`'max-age=31536000; includeSubDomains'`), a `string` overrides it, `false` / omitted leaves it off. `requestId` (the §4.2.1 entity-keyed group, ON by default) sets an `X-Request-ID` (header `REQUEST_ID_HEADER` `'x-request-id'`) AND stashes the id on `context.state` under `REQUEST_ID_STATE` `'requestId'` for downstream logging / correlation; by default it MINTS a fresh `crypto.randomUUID()` and IGNORES an incoming `X-Request-ID` (anti-spoof — the untrusted header is narrowed with `isString`, never `as` — §14), while `requestId.trust` ECHOES a single non-blank incoming id (only safe behind a trusted proxy that strips a forged one; with no incoming id it still mints one). `requestId: false` disables the id entirely. It is MECHANISM, not policy (AGENTS §21) — it ships the conventional secure-default set and stops where an app's own CSP / framing decisions begin (override via the options).
31. **Conditional-GET on dynamic output — weak ETag, reuses the buffer, composes with compression.** [`createETag({ weak?, algorithm? })`](#) adds an `ETag` over a DYNAMIC handler's response body and answers `304 Not Modified` on a matching `If-None-Match` — the dynamic counterpart to [`createStatic`](#)'s file ETag (static files self-ETag from size + mtime). It REUSES the SAME response-buffering machine [`createCompression`](#) uses ([`bufferResponse`](#)) — it does NOT re-implement buffering: it buffers the complete response, hashes the body ([`computeBodyETag`](#)), and either `304`s (no body) when the request's `If-None-Match` equals the tag or sets `ETag` and writes the body unchanged. It acts ONLY on a `GET` whose buffered status is `200`; a `HEAD` / non-GET / non-200 PASSES THROUGH untouched (a `HEAD` has no body to hash — its conditional handling is the static server's, which knows the file's validator). The default validator is WEAK (`W/"<sha256>"`) — encoding-AGNOSTIC, so it survives a downstream content re-encoding (gzip) and still matches on revalidation; `weak: false` emits a STRONG (`"<sha256>"`) byte-identity tag, and `algorithm` overrides the hash (default `DEFAULT_ETAG_ALGORITHM` `'sha256'`). A response that already carries an `ETag` (a handler set its own) is left as-is — its body is written through, never re-hashed (mirroring how compression leaves an already-`Content-Encoding`-set response alone). A STREAMING / SSE response CANNOT be buffered or hashed, so the buffer auto-bypasses and `createETag` leaves it untouched (the same `bufferResponse` pass-through that protects compression). **Composing with [`createCompression`](#)**: both buffer, so ORDER matters — register `createETag` INNER of compression (`server.use(createCompression())` then `server.use(createETag())`). The ETag is then computed over the UNCOMPRESSED representation, rides through compression as a normal header, and compression compresses the body beneath it — so a client gets BOTH the `ETag` header AND a gzip-compressed body, and a later `If-None-Match` revalidation still `304`s (the inner ETag re-derives the same tag from the same uncompressed body; compression passes the bodyless `304` through). Mounting it OUTER of compression would hash the COMPRESSED bytes, coupling the validator to the negotiated coding — wrong. Proven over a real server: the composed stack yields `ETag` + a gzip wire (`readRawResponse`) + a `304` on revalidation, and the composed tag equals the bare-ETag tag (compression did not change what is hashed).

## Observing

The `Server` exposes a typed `emitter` (AGENTS §13) carrying its lifecycle + per-request moments for fire-and-forget observers — logging, metrics, tracing. Subscribe via `server.emitter.on(...)`, or wire initial listeners through the reserved `on?` option; supply an `error?` handler to receive a listener's throw. **Emitting is observation-only**: every event fires strictly AFTER the relevant state transition, so a listener can never change what the server does — and a throwing listener can never corrupt it (the same listener-isolation guarantee as the core Queue / Runner).

```ts
import { createServer } from '@src/server'

const server = createServer({
	on: { start: (port) => console.log(`listening on ${port}`) }, // initial listener
})

server.emitter.on('request', (method, url) => metrics.hit(method, url))
server.emitter.on('error', (error) => log.warn('handler error', error))
```

The `ServerEventMap` vocabulary:

| Event     | Payload              | Fires when                                                                                         |
| --------- | -------------------- | -------------------------------------------------------------------------------------------------- |
| `start`   | `[port]`             | The server bound and is listening (the resolved port).                                             |
| `request` | `[method, url]`      | A request was accepted — its method + pathname (before the chain runs).                            |
| `upgrade` | `[request, handled]` | A protocol upgrade arrived and the fan-out settled — the upgrade request + whether it was claimed. |
| `error`   | `[error]`            | A handler threw past the chain (the server rendered a 500).                                        |
| `stop`    | `[]`                 | `stop` began (the server stopped accepting new requests).                                          |
| `drain`   | `[pending]`          | The drain settled — the count of requests still in flight when it ended (0 on a clean drain).      |

**The listener-isolation safety guarantee.** A listener throw is NEVER allowed to escape into the engine: the emitter isolates it and routes it to its OWN `error` handler (the `error` option, surfaced as `(error, event)`), NOT to a domain event (`error` above is a server-level fault, distinct) — so a buggy observer is isolated yet not silently lost. The `error` handler runs in its own try/catch, so even a throwing handler can't recurse or escape; with no handler, the throw is swallowed silently. Because every emit sits after the relevant transition AND is isolated, a buggy observer **cannot crash the server or corrupt its lifecycle** — proven by the emit-safety test (a throwing `request` observer can't stop the response).

## Streaming (SSE)

The spine's one streaming surface is the **generic Server-Sent-Events seam** — `openSSEStream` (a write handle over a route's response) and `bindStreamingAbort` (a client disconnect → a once-only cancel). It is **agent-agnostic**: it speaks only HTTP + the SSE wire (`SSEMessage`), never `@src/core/agents`, so a live agent turn uses it today and the future **MCP HTTP transport reuses the same seam**. It composes the existing spine (the unified context's raw `response`) and `node:http` — no new primitive.

```ts
import { openSSEStream } from '@src/server'

server.route({
	method: 'GET',
	path: '/clock',
	handler: async (context) => {
		const sse = openSSEStream(context) // writes the SSE headers, flushes them
		for (let n = 0; n < 5 && !sse.closed; n += 1) {
			sse.write({ event: 'tick', data: String(n) }) // event:/data: lines + a blank line
			await new Promise((resolve) => setTimeout(resolve, 1000))
		}
		sse.end()
	},
})
```

**The wire is the exact inverse of the core [`SSEParser`](parsers.md).** Each `write` serializes an `event:` / `id:` / `retry:` line for every present field, then **one `data:` line per `\n`-split segment** of `data` (so a multi-line payload round-trips through the parser's multi-`data` concat: `write({ data: 'a\nb' })` → `data: a` + `data: b` → the parser yields `{ data: 'a\nb' }`), then the terminating blank line — all flushed. `comment(text)` writes a `: text` keep-alive line the parser ignores (no event). Every method is a **safe no-op once `closed`** (the response ended, or the client disconnected), so a late write never throws. A client decodes the response by feeding `response.body`'s bytes through a `TextDecoder({ stream: true })` and `createSSEParser()`.

### The agent-stream route (a consumer pattern, NOT a framework export)

Mapping an `Agent`'s chunk stream onto the seam is a **consumer** concern, so it lives in the application / example layer (and this guide + the live test), **not** in `src/server/http` — that keeps the seam reusable. The route opens the stream, binds a disconnect to `agent.abort()`, pumps each `AgentChunk` as an SSE event, then emits the settled result:

```ts
import { bindStreamingAbort, openSSEStream } from '@src/server'
import type { AgentChunk, AgentInterface, RouteHandlerContextInterface } from '@src/core'

function mapChunkToSSE(chunk: AgentChunk): { event: string; data: string } {
	if (chunk.type === 'token') return { event: 'token', data: chunk.content }
	if (chunk.type === 'tool')
		return { event: 'tool', data: JSON.stringify({ call: chunk.call, result: chunk.result }) }
	return { event: 'usage', data: JSON.stringify(chunk.usage) }
}

async function streamAgent(
	context: RouteHandlerContextInterface,
	agent: AgentInterface,
): Promise<void> {
	const stream = agent.stream()
	const cleanup = bindStreamingAbort(context, (reason) => stream.abort(reason)) // disconnect → abort
	const sse = openSSEStream(context)
	try {
		for await (const chunk of stream.events) sse.write(mapChunkToSSE(chunk))
		sse.write({ event: 'result', data: JSON.stringify(await stream.result) }) // final outcome
	} catch (error) {
		sse.write({ event: 'error', data: error instanceof Error ? error.message : String(error) })
	} finally {
		sse.end()
		cleanup() // detach the disconnect listeners on a normal end — no leak
	}
}
```

The `AgentChunk` → SSE mapping: a `token` → `{ event: 'token', data: content }`, a `tool` → `{ event: 'tool', data: JSON.stringify({ call, result }) }`, a `usage` → `{ event: 'usage', data: JSON.stringify(usage) }`, the settled outcome → `{ event: 'result', data: JSON.stringify(result) }`, and an error → `{ event: 'error', data: message }`.

### Disconnect → once-only abort

`bindStreamingAbort(context, onAbort)` wires the request's `'close'` / `'aborted'` and the response's `'close'`; the **first** to fire calls `onAbort` **exactly once** (a later one is ignored), and the returned cleanup removes every listener it added. Call the cleanup when the stream ends normally so a subsequent natural `'close'` does not fire `onAbort` and nothing leaks. This is what turns a browser closing the EventSource into `agent.abort()`, committing the turn's partial.

### Graceful-drain interaction

A long-lived SSE response is an **in-flight request** (its `finish` thunk decrements the drain counter when the response actually ends — on `sse.end()` or the socket closing). So an open stream must not wedge [the graceful drain](#) (`server.stop()`) forever. Two mechanisms keep `stop()` bounded:

1. **Clean path (recommended).** A consumer observes the server's `stop` event and ends the stream — `server.emitter.on('stop', () => stream.abort())` (or `sse.end()`). The stop → abort → the `for await` loop ends → `sse.end()` → the response finishes → the request drains → `stop()` resolves **promptly**, well under the drain deadline. (The seam stays agent-agnostic; this wiring is the consumer's.)
2. **Backstop.** Even without that wiring, `stop()` arms the drain `Timeout`; if the stream has not self-terminated by the deadline the server **force-closes** the socket (`closeAllConnections`), the response `'close'` fires → `bindStreamingAbort` fires `onAbort` → the producer cancels → the stream ends. So `stop()` **always resolves** (at worst at the deadline) — it never hangs. This is the same bounded force-close the spine already uses for a hung handler.

Both are pinned by tests: the deterministic seam test proves an open stream + `server.stop()` resolves (no hang) via the clean path, and the existing hung-handler drain test proves the backstop.

## The upgrade seam

The spine's one connection-handoff surface is the **generic upgrade seam** — `upgrade(handler)`. A raw `node:http` protocol upgrade (an HTTP request carrying `Connection: Upgrade`) is fanned out to the registered `UpgradeHandler`s; the first to **claim** the socket owns the raw connection from there. It is **protocol-agnostic**: the spine knows nothing of WebSocket (or any upgrade protocol) — it hands the claimer the raw `node:stream` Duplex socket and gets out of the way, so a [WebSocket transport](websocket.md) (built on `createNodeWebSocket`) attaches THROUGH this seam without the spine learning RFC 6455.

```ts
import { createNodeWebSocket, createServer } from '@src/server'

const server = createServer()
server.upgrade((request, socket, head) => {
	// Only claim the paths this transport owns — decline (return false) otherwise,
	// so another registered handler (or the spine's destroy) handles the rest.
	if (request.url !== '/ws') return false
	// Claim it: build the WebSocket over the raw socket (the wrapper writes the 101
	// handshake from the client key) and return true — we now own this connection.
	const ws = createNodeWebSocket({
		socket,
		key: request.headers['sec-websocket-key'],
		head,
		on: { message: (text) => ws.send(`echo: ${text}`) },
	})
	return true
})

const port = await server.start()
```

**The fan-out + claim contract.** An `UpgradeHandler` is `(request, socket, head) => boolean`. The spine walks the registered handlers in registration order and:

- The **first** handler that returns `true` **claims** the socket — it now OWNS the connection (it writes the `101` and speaks its wire protocol), **no later handler runs**, and the spine never touches the socket again.
- A handler returning **falsy declines**, so the **next** registered handler gets a chance (a path-scoped transport returns `false` for paths it does not own).
- A handler that **throws is caught and treated as a decline** — the throw is surfaced on the `error` event (the same channel an escaping route-handler throw uses, see [a handler throw never crashes the server](#)), and the fan-out continues to the next handler. The `'upgrade'` listener runs raw on `node:http`, so a bare uncaught throw here would be an `uncaughtException` that crashes the whole process and leaks the socket; catching it keeps the seam **symmetric with the request path** — a buggy handler can never take the server down.
- If **no** handler claims it (none registered, every one declined, or every one threw), the spine **destroys** the un-upgraded socket — an unhandled upgrade never leaks a dangling connection.

The raw triple is node's own `IncomingMessage` / `Duplex` / `Buffer` — the upgrade `request` (its headers carry `Upgrade` / `Sec-WebSocket-Key`), the upgraded Duplex `socket`, and any `head` bytes buffered after the upgrade request (replay them through the claimer's parser). They cross the boundary narrowed by node's types, never asserted (§14).

**Observability.** When an upgrade settles, the `Server` fires an `upgrade` event — `[request, handled]` — carrying the upgrade request and whether a handler claimed it. Like every server event it is observation-only: it fires **strictly after** the fan-out resolves, so a listener can neither claim the socket nor change the decision, and the emitter isolates a throwing one (routing it to the `error` handler — see [Observing](#observing)).

**Lifecycle.** The `'upgrade'` listener is wired in `start()` onto that run's underlying `node:http` server, which `stop` / `destroy` discard — so it is collected with the closed server and never leaks across a restart (the same per-run lifecycle as the request handler).

### The WebSocket transport (a consumer pattern, NOT a framework export)

Mapping a claimed upgrade onto a message protocol (WebSocket → MCP / JSON-RPC) is a **consumer** concern, so — like the [agent-stream route](#the-agent-stream-route-a-consumer-pattern-not-a-framework-export) and the [S2 proxy route](#the-s2-proxy-route-a-consumer-pattern-not-a-framework-export) — it lives in the transport / application layer, NOT in `src/server/http`. The seam keeps the spine reusable: it speaks only HTTP + the raw socket, so the same `upgrade(handler)` serves a WebSocket transport, a future raw-TCP protocol, or anything else that negotiates over an HTTP upgrade.

## Tokens

The spine ships its own **S2 stateless-token gate** — `createTokenGuard` over the `signToken` / `verifyToken` primitives. A token is fully self-describing (`<base64url-payload>.<base64url-signature>`, the payload a JSON `{ value, exp? }`, the signature its HMAC-SHA256 under a secret), so there is no server-side session store: a verifier needs only the secret. It is the layer a browser app's obfuscated bearer lands on before reaching the agent runtime.

```ts
import {
	createErrorBoundary,
	createServer,
	createTokenGuard,
	signToken,
	TOKEN_STATE,
} from '@src/server'

// Supply a strong secret from secure config — never a hardcoded fallback.
const SECRET = process.env.TOKEN_SECRET
if (SECRET === undefined) throw new Error('TOKEN_SECRET must be set')

const server = createServer()
server.use(createErrorBoundary()) // renders the guard's 401
server.use(createTokenGuard({ secret: SECRET }))
server.route({
	method: 'GET',
	path: '/me',
	handler: (context) => context.json({ subject: context.state.get(TOKEN_STATE) }),
})

// The client sends the signed token as a Bearer:
//   fetch(url, { headers: { authorization: `Bearer ${signToken('alice', { secret: SECRET, ttl: 3_600_000 })}` } })
```

- **Mint with `signToken(value, { secret, ttl? })`.** The `value` is any opaque string (a session id, a deployment marker). A `ttl` (ms) binds an expiry INSIDE the signed payload, so it is HMAC-covered — a client cannot lengthen it without invalidating the signature.
- **Always set a `ttl` for a browser-bound bearer.** A token minted with NO `ttl` never expires, and because the scheme is stateless there is no per-token revocation — the ONLY way to invalidate an issued token is to rotate the secret, which invalidates EVERY outstanding token at once. So any bearer that travels to a browser should carry a `ttl` (so a leaked token self-expires); reserve a non-expiring token for a trusted, server-held marker you can afford to invalidate only by a full secret rotation.
- **A blank secret is rejected (fail-closed).** Blank / whitespace-only secrets are IGNORED, so a secret with no usable entry (`''`, `'   '`, an empty or all-blank rotation list) makes `signToken` throw and `verifyToken` return `undefined` rather than signing / verifying under a publicly-computable empty-key HMAC. A blank entry alongside a real secret in a rotation list is simply dropped — the real secret still works.
- **Verify with `verifyToken(token, secret)` — or behind `createTokenGuard`.** It returns the embedded value when the token is valid + unexpired, else `undefined`. It is **total**: it never throws on adversarial input (a malformed token, a forged signature, a hostile payload, an expired token all return `undefined`), so it is safe on untrusted request input. The signature compare is length-guarded + constant-time (`timingSafeEqual`).
- **Rotate secrets with a list.** A `TokenSecret` may be a `[current, ...older]` list: `signToken` signs with the FIRST (current) secret, while `verifyToken` / `createTokenGuard` accept a token signed by ANY secret in the list. Rotate by prepending the new secret and keeping the old until every outstanding token has expired — zero downtime.
- **`createTokenGuard` rejects to a `401`.** A missing token throws `HTTPError(401, 'missing token')`; an invalid / expired / tampered one throws `HTTPError(401, 'invalid token')` — both rendered by `createErrorBoundary` (register it OUTSIDE the guard), so the failure path never writes the response directly. On success the verified value is on `context.state` under `TOKEN_STATE` for downstream middleware + the handler.
- **`header` / `scheme` are configurable.** Defaults read `Authorization: Bearer <token>` (`DEFAULT_TOKEN_HEADER` / `DEFAULT_TOKEN_SCHEME`); set `scheme: ''` to treat the whole header value as the raw token, or `header` to read a custom header (matched lower-cased, as node delivers it).

### The S2 proxy route (a consumer pattern, NOT a framework export)

The headline S2 use is a **proxy route**: a browser holds only a short-lived, obfuscated bearer (signed with the server's secret) and never the real LLM API key; the app's guarded route validates that bearer, then injects the REAL key server-side and forwards to the real model. Like the [agent-stream route](#the-agent-stream-route-a-consumer-pattern-not-a-framework-export), this mapping is **application / deployment policy**, so it lives in the app layer (and this guide + the live test), NOT in `src/server/http` — the guard + token primitives stay generic.

```ts
import { createErrorBoundary, createServer, createTokenGuard } from '@src/server'

// Supply a strong secret from secure config — never a hardcoded fallback.
const SECRET = process.env.TOKEN_SECRET
if (SECRET === undefined) throw new Error('TOKEN_SECRET must be set')

// The upstream is a FIXED, server-controlled constant — NEVER derived from request
// input (see the SSRF note below).
const UPSTREAM = 'https://api.example-llm.com/v1/chat'

const server = createServer()
server.use(createErrorBoundary()) // renders the guard's 401
server.use(createTokenGuard({ secret: SECRET }))
server.route({
	method: 'POST',
	path: '/api/chat',
	handler: async (context) => {
		// The guard already validated the browser's bearer (a 401 short-circuits here).
		const body = await context.body()
		const upstream = await fetch(UPSTREAM, {
			method: 'POST',
			// Inject the REAL key server-side — the browser never holds it.
			headers: {
				'content-type': 'application/json',
				authorization: `Bearer ${process.env.LLM_KEY}`,
			},
			body: JSON.stringify(body),
		})
		context.json(await upstream.json())
	},
})
```

The browser-side provider points at this route and supplies the obfuscated bearer through its own transport hook (e.g. the Ollama provider's `headers: () => ({ authorization: 'Bearer ' + signToken(...) })`). A valid bearer round-trips the upstream model's response; an invalid one is a `401` that never reaches the upstream — proven deterministically (a stand-in upstream server) and live (against the real model) in the tests below.

**Harden the proxy — SSRF + authenticated passthrough.** Two responsibilities are the consumer's, not the guard's:

- **Keep the upstream target server-controlled — NEVER derive it from request input.** The forwarded URL must be a FIXED constant (as `UPSTREAM` above) or validated against a host allow-list. If the upstream host/path/port is taken from the request body, query, or a header, the proxy becomes a Server-Side Request Forgery vector ([CWE-918](https://cwe.mitre.org/data/definitions/918.html)): an attacker steers the server — which is holding the REAL key — at an arbitrary host, exfiltrating that injected credential or reaching internal services the browser never could.
- **Treat the forwarded body as an authenticated passthrough.** Once the guard passes, the request rides on the server's OWN upstream credential, so a valid bearer can drive any request the body expresses (any model, any parameters, unbounded spend). The token proves only that the caller holds a valid bearer — it does NOT scope what they may ask for. Per-tenant scoping, model allow-listing, and spend / rate caps on the forwarded body are the consumer's responsibility, enforced server-side before the upstream `fetch`.

## Patterns

### A minimal server

```ts
import { createServer } from '@src/server'

const server = createServer()
server.route({ method: 'GET', path: '/health', handler: (context) => context.json({ ok: true }) })
const port = await server.start()
// ... later
await server.stop()
```

### The middleware order (outer → inner)

```ts
import { createBodyParser, createCors, createErrorBoundary, createServer } from '@src/server'

const server = createServer()
// Register OUTER-first: the error boundary surrounds everything, then CORS, then the parser.
server.use(createErrorBoundary()) // 1. catches any downstream throw → status response
server.use(createCors()) // 2. sets cross-origin headers / answers preflight
server.use(createBodyParser({ limit: 1_000_000 })) // 3. parses + size-limits the body
server.route({
	method: 'POST',
	path: '/users',
	handler: async (context) => context.json(await context.body(), 201),
})
```

The onion runs `errorBoundary → cors → bodyParser → handler` and unwinds in reverse — so the error boundary sees a throw from anywhere downstream, and a body over the limit becomes a `413`.

### Path params, query, and the state bag

```ts
const server = createServer()
server.use(async (context, next) => {
	context.state.set('db', database) // thread an app dependency to the handler
	await next()
})
server.route({
	method: 'GET',
	path: '/users/:id',
	handler: async (context) => {
		const db = context.state.get('db')
		const sort = context.query.get('sort')
		context.json({ id: context.params.id, sort })
	},
})
```

### Client-facing errors

```ts
import { createErrorBoundary, createServer, HTTPError } from '@src/server'

const server = createServer()
server.use(createErrorBoundary())
server.route({
	method: 'GET',
	path: '/users/:id',
	handler: async (context) => {
		const user = await find(context.params.id)
		if (user === undefined) throw new HTTPError(404, 'user not found') // → 404 { error: 'user not found' }
		context.json(user)
	},
})
```

### Route groups

```ts
const server = createServer()
const api = server.group('/api')
api.add({ method: 'GET', path: '/users/:id', handler: getUser }) // → GET /api/users/:id
const v2 = api.group('/v2')
v2.add({ method: 'GET', path: '/users/:id', handler: getUserV2 }) // → GET /api/v2/users/:id
```

### Rate-limit per authenticated client

The rate limiter's default client key prefers the verified token an upstream `createTokenGuard` stashed under `TOKEN_STATE` (falling back to the socket peer IP), so registering the guard **before** the limiter buckets requests per authenticated subject rather than per shared IP — the two primitives compose by ordering alone:

```ts
import { createErrorBoundary, createRateLimiter, createServer, createTokenGuard } from '@src/server'

const secret = process.env.TOKEN_SECRET
if (secret === undefined) throw new Error('TOKEN_SECRET must be set')

const server = createServer()
server.use(createErrorBoundary()) // renders the guard's 401
server.use(createTokenGuard({ secret })) // stashes the verified token on state...
server.use(createRateLimiter({ max: 100, window: 60_000 })) // ...which keys the limit (per subject, not per IP)
server.route({ method: 'GET', path: '/', handler: (context) => context.text('ok') })
```

The over-limit request is a `429` with a `Retry-After` header (whole seconds until the window resets); it short-circuits **without** throwing, so no error boundary is required for the limit itself. Behind a trusted proxy, pass an explicit `key` rather than trusting `X-Forwarded-For` (it is spoofable).

### Security headers + conditional GET

`createSecurity` sets a configurable secure-default header set (and an `X-Request-ID` for log correlation); `createETag` adds a content `ETag` + conditional `304` to dynamic GET output. The default header set is hardened: the CSP carries `base-uri 'self'` / `object-src 'none'` / `frame-ancestors 'self'` / `form-action 'self'` beside `default-src 'self'` (a custom `csp` replaces it wholesale), and it ships `Cross-Origin-Opener-Policy: same-origin` + `Cross-Origin-Resource-Policy: same-origin` (process isolation + embed-blocking; COEP `require-corp` is opt-in via `coep` since it breaks cross-origin subresources). The `X-Request-ID` is correlation-safe: a trusted incoming id is honored only when it matches a strict `^[A-Za-z0-9_-]{1,200}$` charset (else a fresh UUID is minted), so a CR/LF / control / oversized client value never rides into a response header or the logged `state`. `createETag`'s `If-None-Match` uses the RFC 7232 §2.3.2 **weak** comparison — `W/"x"` matches `"x"`, `*` matches any current representation, and a comma-list matches if any member does. Register `createSecurity` near the top, and `createETag` **inner** of `createCompression` so the ETag is computed over the uncompressed body — the client then gets the ETag header **and** a gzip body, and a revalidation still `304`s:

```ts
import {
	createCompression,
	createETag,
	createSecurity,
	createServer,
	REQUEST_ID_STATE,
} from '@src/server'

const server = createServer()
server.use(createSecurity({ hsts: true })) // secure-default headers + X-Request-ID on state (HSTS behind HTTPS)
server.use(createCompression()) // OUTER of ETag — compresses the body
server.use(createETag()) // INNER — weak ETag over the UNCOMPRESSED body, conditional 304
server.route({
	method: 'GET',
	path: '/data',
	handler: (context) => {
		context.response.setHeader('X-Trace', String(context.state.get(REQUEST_ID_STATE))) // correlate logs
		context.json(largePayload) // → ETag + gzip; a matching If-None-Match → 304 (no body)
	},
})
await server.start()
```

`createSecurity` is opt-out per header — override a value with a string, suppress it with `false`. HSTS is **off by default** (it is HTTPS-only and browser-pinned). The `X-Request-ID` defaults to a fresh `crypto.randomUUID()` and **ignores** an incoming header (anti-spoof); behind a trusted proxy that strips a forged one, set `requestId: { trust: true }` to echo the upstream id. `createETag` only ETags a cacheable `GET 200` — a `HEAD` / non-GET / non-200 and any **streaming / SSE** response pass through untouched (an unbounded stream can't be buffered or hashed); it respects a handler's own `ETag`. The recommended order is compression **outer**, ETag **inner** — reversing them would hash the compressed bytes, wrongly coupling the validator to the negotiated coding.

### Add sessions

```ts
import { createServer, createSession, SESSION_STATE, Session } from '@src/server'

const server = createServer()
server.use(createSession()) // auto-session: mints a Session on a missing id, sets the session-id header
server.use((context, next) => {
	// Every request now carries a Session on context.state; read it back, narrow with instanceof.
	const session = context.state.get(SESSION_STATE)
	if (session instanceof Session) session.data.set('lastSeen', Date.now()) // per-session app state
	return next()
})
server.route({ method: 'GET', path: '/me', handler: (context) => context.empty(204) })
await server.start()
```

`createSession()` is the generic session middleware — it mints a `Session` whenever a request carries no valid `session-id`, echoes the minted id in the response header, and stashes the resolved session on `context.state` under `SESSION_STATE` (read it back with `instanceof Session`, never `as`). Tune it: `ttl` lazily evicts an idle session (no background timer); `require: true` + a custom `mint` reject an unsessioned request via `onMissing` instead of auto-minting; `header` / `key` / `create` swap the wire header, the state key, and the session entity. A `DELETE` carrying a valid id ends the session (`204`). The per-id storage is a pluggable [`SessionStoreInterface`](#sessionstoreinterface) (the lean session analogue of [`QueueStoreInterface`](workers.md) — `get` / `set` / `delete`); it defaults to an in-memory [`createMemorySessionStore`](#) (the lifted closure store, the same lazy-TTL shape as [`createRateLimiter`](#)), and a durable / DB-backed `store` replaces it without touching `createSession` — or one `createMemorySessionStore()` is shared across several middlewares. It is generic over the session type — the MCP transport's [`createMCPSession`](mcp.md) builds on it. For a browser app that prefers a cookie over a header, [`createCookieSession`](#add-cookie-sessions--csrf) swaps the id transport (and adds `regenerate` / `destroy` + a way-out data persist); pair it with [`createCSRFGuard`](#add-cookie-sessions--csrf) for double-submit CSRF.

### Add cookie sessions + CSRF

For a browser app, carry the session id in a signed cookie instead of a header — `createCookieSession` is `createSession` with the cookie transport (the same store / mint / TTL machine), plus `regenerate()` (rotate the id, anti-fixation) and `destroy()` (logout) via the control handle on `context.state`, and a way-out `store.set` so a durable store round-trips the session's `data`:

```ts
import {
	createCookieSession,
	createCSRFGuard,
	createBodyParser,
	createErrorBoundary,
	createServer,
	CSRF_STATE,
	SESSION_CONTROL_STATE,
	SESSION_STATE,
	Session,
	isSessionControl,
} from '@src/server'

// Supply strong secrets from secure config — never hardcoded fallbacks.
const SESSION_SECRET = process.env.SESSION_SECRET
const CSRF_SECRET = process.env.CSRF_SECRET
if (SESSION_SECRET === undefined || CSRF_SECRET === undefined)
	throw new Error('secrets must be set')

const server = createServer()
server.use(createErrorBoundary()) // renders the CSRF 403
server.use(createBodyParser()) // so a form-field CSRF token is readable
server.use(
	createCookieSession({ secret: SESSION_SECRET, ttl: 86_400_000, cookie: { secure: true } }),
)
server.use(createCSRFGuard({ secret: CSRF_SECRET }))

// A safe GET: the session rides a signed cookie; expose the CSRF token for the page to echo.
server.route({
	method: 'GET',
	path: '/account',
	handler: (context) => {
		const session = context.state.get(SESSION_STATE)
		if (session instanceof Session) session.data.set('seen', Date.now())
		context.json({ csrf: context.state.get(CSRF_STATE) })
	},
})

// A mutating POST: the CSRF guard verifies the echoed token; rotate the id on privilege change.
server.route({
	method: 'POST',
	path: '/account',
	handler: (context) => {
		const control = context.state.get(SESSION_CONTROL_STATE)
		if (isSessionControl(control)) control.regenerate() // anti-fixation after a login / role change
		context.empty(204)
	},
})
await server.start()
```

`createCookieSession({ secret })` signs the id into a `Set-Cookie` (default name `session`, `HttpOnly`, `SameSite=Lax`; set `cookie.secure` for direct HTTPS) and reads it back from the `Cookie` header — a tampered cookie reads as no session and re-mints. Every `createSession` knob (`ttl` / `key` / `create` / `mint` / `require` / `onMissing` / `store`) applies; supply a durable `store` (`createDatabaseSessionStore`) to persist sessions across restarts. The handler flags `regenerate()` (id rotation, data preserved) or `destroy()` (logout, cookie cleared) on the control handle under `SESSION_CONTROL_STATE` — the middleware does the storage + cookie work. `createCSRFGuard({ secret })` mints a readable (`HttpOnly:false`, `SameSite=Strict`) signed token cookie on a safe method and exposes the raw token under `CSRF_STATE`; on a mutating method it verifies the submitted token (the `x-csrf-token` header or the `_csrf` body field) against the signed cookie, rejecting a missing / invalid / mismatched token with a `403`. When a session is on `state` (mount the guard AFTER the session middleware), the token is additionally **bound to that session** — the value is `signToken(sessionId)`, so a mutating request whose token resolves to a different session id is rejected and a token minted under one session cannot be replayed under another; sessionless, it falls back to a signed-random double-submit. It keeps no CSRF store of its own and is pure mechanism — pair it with the session above; the client reads the token cookie (or the exposed value) and echoes it on every mutating request.

### Serve a built SPA in front of a JSON / SSE API

`createStatic` serves a directory of files (server-only `node:fs`) with an airtight traversal guard, MIME types, a weak ETag + conditional `304`, HTTP `Range`, a directory `index`, and an optional **SPA document fallback**. It is **composable** — a request that does not resolve to a served file falls through `next()`, so it sits IN FRONT of the API routes and only the unmatched, client-router routes get the SPA shell:

```ts
import { createServer, createStatic } from '@src/server'

const server = createServer()

// API routes first — they are matched by the router; createStatic falls through to them.
server.route({ method: 'GET', path: '/api/health', handler: (c) => c.json({ ok: true }) })

// Then the static mount: serve ./public, cache assets for an hour, and fall back to the
// index document for client routes (`/users/42`) so a client-side router owns them — but
// NOT for `/api/*` (it 404s honestly) or a missing extension'd asset (`/app.js` → 404).
server.use(createStatic({ root: '/srv/app/public', maxAge: 3600, fallback: true }))

const port = await server.start()
// GET /app.js        → the file (text/javascript), ETag + Accept-Ranges
// GET /users/42      → index.html (the SPA shell; Accept: text/html, extension-less)
// GET /api/health    → { ok: true } (the route; the /api exclude blocks the fallback)
```

- **The traversal guard is airtight.** A `../`, an encoded `..` (`%2e%2e`), an absolute path, a NUL byte, or a sibling sharing the root's string prefix never resolves inside `root` — it is refused (the file is never read). A **Windows reserved DOS device name** segment (`CON` / `COM1` / `NUL` / `CON.txt` / `AUX.` / the superscript `COM¹`, CVE-2025-27210) is also refused — a device alias resolves UNDER `root` by string, so the containment check alone would pass it. It is the highest-risk piece; it is unit-tested directly and proven over the wire (an attack body never contains a secret living outside `root`, and a device path is never opened).
- **`Range` / `206` / `416`.** A `Range: bytes=start-end` (closed / open / suffix) streams a `206` with `Content-Range` + the exact slice; an unsatisfiable range is a `416`. `Accept-Ranges: bytes` advertises support.
- **Conditional `304`.** A weak `ETag` (`W/"<size>-<mtimeMs>"`, default on; `etag: false` disables) lets a client revalidate — a matching `If-None-Match` returns `304` with no body.
- **Dotfile policy.** `dotfiles: 'ignore'` (default — a `.env` falls through), `'deny'` (`403`), or `'allow'` (serve).
- **`HEAD`** serves the headers (including `Content-Length`) with no body; a non-`GET`/`HEAD` method falls through.
- **It is mechanism, not the Filesystem abstraction.** It is a plain middleware over `node:fs` — there is no browser static-serving peer, so no cross-env interface (it is NOT the deferred Filesystem / Sandbox).

### Composing the full server

Every battery composes on ONE server. The recommended production order — outer → inner — stacks all of them so a single JSON response carries compression **and** the security headers **and** an `ETag` **and** both session + CSRF cookies **and** CORS at once, a thrown handler error is caught **and** still compressed, a mutating request is CSRF-protected, and a built SPA is served in front of the routes:

```ts
import {
	createBodyParser,
	createCompression,
	createCookieSession,
	createCors,
	createCSRFGuard,
	createErrorBoundary,
	createETag,
	createRateLimiter,
	createSecurity,
	createServer,
	createStatic,
} from '@src/server'

// Supply strong secrets from secure config — never hardcoded fallbacks.
const sessionSecret = process.env.SESSION_SECRET
const csrfSecret = process.env.CSRF_SECRET
if (sessionSecret === undefined || csrfSecret === undefined) throw new Error('secrets must be set')

const server = createServer()
server.use(createCompression()) // outermost — compresses EVERY downstream body, incl. error bodies
server.use(createErrorBoundary()) // catches a downstream throw → status (its body, being inner, still compresses)
server.use(createSecurity({ hsts: true })) // secure-default headers + X-Request-ID, early (HSTS behind HTTPS)
server.use(createETag()) // content ETag over the UNCOMPRESSED body (inner of compression) + conditional 304
server.use(createCors()) // cross-origin headers / preflight, early
server.use(createRateLimiter({ max: 100, window: 60_000 })) // per-client window (keys per token if a guard precedes it)
server.use(createBodyParser()) // parse + size-limit (so a form-field CSRF token is readable)
server.use(
	createCookieSession({ secret: sessionSecret, ttl: 86_400_000, cookie: { secure: true } }),
) // signed-cookie session
server.use(createCSRFGuard({ secret: csrfSecret })) // double-submit CSRF (mutating → verify; emits its OWN cookie)
server.use(createStatic({ root: '/srv/app/public', fallback: true })) // serve the SPA, FALL THROUGH (next()) to routes
server.route({ method: 'GET', path: '/api/me', handler: (context) => context.json({ ok: true }) })
await server.start()
```

The one-line rationale per non-obvious placement:

- **`createCompression` outermost** — it wraps every downstream write, so it compresses the route bodies AND the error-boundary's error body (a boundary OUTSIDE compression would leave error responses uncompressed). It auto-bypasses a stream / SSE response untouched.
- **`createErrorBoundary` just inside it** — it catches any throw from everything below into a status response; sitting INNER of compression, that error body is buffered + compressed too. (The `Server` spine is the last-resort guard — an escaping throw it never saw still becomes a `500` so the process can't crash — but the boundary is what renders a _structured_ error.)
- **`createSecurity` / `createCors` early** — header-setters belong near the top so every response (success, error, 304, static) carries the secure-default headers + the CORS allow-origin.
- **`createETag` INNER of compression** — the `ETag` is hashed over the UNCOMPRESSED body, so a revalidation `304`s regardless of the negotiated coding; reversing them would hash the compressed bytes and wrongly couple the validator to gzip.
- **`createRateLimiter` before the body parser** — reject an over-quota request with a `429` before spending work parsing its body; place it AFTER a `createTokenGuard` (when present) so it buckets per authenticated subject rather than per shared IP.
- **`createBodyParser` before the session + CSRF** — the body is parsed once (cached) so the CSRF guard can read a `_csrf` form field, and the session middleware runs ahead of the guard it pairs with.
- **`createCookieSession` before `createCSRFGuard`** — establish the session (its signed cookie) first; the CSRF guard then mints its OWN cookie alongside it (both `Set-Cookie`s survive compression — the multi-cookie buffer contract).
- **`createStatic` before the routes** — it is composable (`next()` on no match), so it serves built assets + the SPA shell and falls through to the API routes for everything it doesn't own.

This exact stack is proven end-to-end over a real server in [`tests/src/server/http/composition.test.ts`](../../tests/src/server/http/composition.test.ts).

### Practices

- **Register the error boundary first** — it must be the outermost middleware to surround every downstream link and the handler.
- **Throw `HTTPError` for client faults** — its `status` + `message` become the response; a bare `throw` is a hidden 500.
- **Read the body once via `context.body()`** — it caches, so a body-parser middleware and the handler share one read; never re-read the raw `request` stream.
- **`stop()` for graceful shutdown, `destroy()` for teardown** — `stop` drains in-flight requests up to the `drain` deadline; `destroy` force-closes everything.
- **Observe, don't drive** — subscribe to `server.emitter` for lifecycle moments (see [Observing](#observing)); emitting is a pure side-channel, so a listener never changes what the server does (and a throwing one can't crash it).
- **Guard with a token, never a session store** — `createTokenGuard` over the stateless `signToken` / `verifyToken` HMAC tokens (see [Tokens](#tokens)); register the error boundary outside it so a `401` renders. Read the verified value off `context.state.get(TOKEN_STATE)`. Rotate secrets with a `[current, ...older]` list, never a redeploy.
- **Add sessions with `createSession`** — mount `server.use(createSession())` for the generic auto-session (mint-on-missing → `session-id` header, idle-TTL, `DELETE`-end), and read the resolved session off `context.state.get(SESSION_STATE)` (narrow with `instanceof Session`, never `as`); set `require: true` + a custom `mint` to reject an unsessioned request rather than auto-minting. It is generic over the session type, so a transport (MCP) builds its own session layer on it rather than re-implementing the store. The storage is a pluggable `SessionStoreInterface` (default in-memory via `createMemorySessionStore`); supply a durable `store` to persist sessions across restarts (or share one store across middlewares) without touching `createSession`.
- **For a browser, prefer `createCookieSession` + `createCSRFGuard`** — carry the id in a signed cookie (the same store / lifecycle), `regenerate()` the id on a privilege change (anti-fixation), `destroy()` to log out (clears the cookie). Always set `cookie: { secure: true }` for a direct-HTTPS deployment and a `ttl` so the cookie self-expires. Pair it with `createCSRFGuard` for mutating requests (register the error boundary outside it so the `403` renders; mount `createBodyParser` ahead if the token rides a form field) — the client echoes the exposed `CSRF_STATE` token (or reads the non-`HttpOnly` cookie) on every POST / PUT / PATCH / DELETE.
- **Serve static assets with `createStatic`, mounted AFTER the API routes** — it is composable (it falls through `next()` on no match), so register the API routes first and let it serve the rest. Enable `fallback: true` to hand client-router routes the SPA shell while `/api/*` and missing extension'd assets 404 honestly; set `maxAge` for asset caching. Trust the traversal guard — never hand-roll path resolution. It is a plain `node:fs` middleware, not the deferred Filesystem abstraction.

## Security

The spine is **secure by default**: mount the edge middleware and the strong setting is already on — `createSecurity()` ships hardened headers, `createCookieSession` / `createCSRFGuard` ship `Secure`-aware signed cookies, `createBodyParser` caps a zip-bomb, `createStatic` guards traversal, `createMultipart` trusts the bytes not the declared type. The defaults are deliberately strict; this section is the developer-facing CONSOLIDATION — what is on, WHY it is the safe choice, and the EXACT knob to loosen it for your situation (with the trade-off you accept). Every default below is the real value from [`constants.ts`](../../src/server/http/constants.ts); every override is the real option from [`types.ts`](../../src/server/http/types.ts). The deep behavioral guarantees live in the [Contract](#contract) clauses (CORS 13, tokens 17, rate limit 18, sessions 20–22, cookie sessions 23, static 24, CSRF 25, decompression 27, multipart 29, security headers 30) — this section is the override map over them.

### Security headers (`createSecurity`)

`createSecurity()` sets a conventional secure-default header set then `next()`s (the set-header-then-continue shape of `createCors`). Mount it near the TOP of the chain so EVERY response — success, error, `304`, static — carries the headers. Each header is opt-OUT: a value-bearing one takes `string` (override the value wholesale) `| false` (omit it) — the [`resolveSecurityHeader`](#) "value, or off" union (AGENTS §4.4), not a behavioral toggle.

- **CSP (`csp`)** — default [`DEFAULT_CSP`](#) = `default-src 'self'; base-uri 'self'; object-src 'none'; frame-ancestors 'self'; form-action 'self'`. `default-src 'self'` loads resources only from the page's own origin; the four added directives close gaps `default-src` alone misses — `base-uri 'self'` (an injected `<base href>` can't re-point relative URLs at an attacker), `object-src 'none'` (no `<object>`/`<embed>` plugin XSS surface), `frame-ancestors 'self'` (the modern clickjacking defense, supersedes `X-Frame-Options` in CSP-aware browsers), `form-action 'self'` (an injected `<form>` can't POST credentials off-origin). **Loosen:** most apps tighten/extend it for a CDN or inline scripts — `csp: "default-src 'self'; script-src 'self' 'unsafe-inline' https://cdn.example.com"` REPLACES the default WHOLESALE (it is not merged, so re-state every directive you still want); `csp: false` omits the header entirely (you forfeit injection + clickjacking hardening). Prefer a nonce/hash over `'unsafe-inline'`.
- **`X-Content-Type-Options: nosniff`** — UNCONDITIONAL (no value to tune, no downside), so it is not an option. It stops a browser MIME-sniffing a response into an executable type.
- **`X-Frame-Options` (`frame`)** — default [`DEFAULT_FRAME_OPTIONS`](#) = `DENY` (the page may NEVER be framed — the strictest clickjacking defense, for older clients that predate CSP `frame-ancestors`). **Loosen:** `frame: 'SAMEORIGIN'` (your own origin may frame it — e.g. an in-app preview); `frame: false` omits it (rely on CSP `frame-ancestors` alone).
- **COOP (`coop`)** — default [`DEFAULT_COOP`](#) = `same-origin`. Cross-Origin-Opener-Policy process-ISOLATES the browsing-context group, severing the `window.opener` / named-window channel a cross-origin attacker (or an XS-Leak / Spectre side-channel) would reach the page through. **Trade-off:** `same-origin` BREAKS an OAuth / payment **popup** that talks back via `window.opener`. **Loosen:** `coop: 'same-origin-allow-popups'` (keep the isolation but let YOUR popups retain the opener reference); `coop: false` omits it.
- **CORP (`corp`)** — default [`DEFAULT_CORP`](#) = `same-origin`. Cross-Origin-Resource-Policy blocks ANOTHER origin from `<img>`/`<script>`/`fetch`-embedding this response (cross-origin resource inclusion + speculative-read defense). **Trade-off:** `same-origin` blocks a public CDN / shared-asset / cross-origin API use. **Loosen:** `corp: 'cross-origin'` (anyone may embed it — a public CDN asset) or `corp: 'same-site'` (a sibling subdomain); `corp: false` omits it. (CORP is NOT COEP — see below.)
- **COEP (`coep`)** — **OFF by default** (a BOOLEAN opt-IN, not part of the secure-default set). Cross-Origin-Embedder-Policy `require-corp` (the [`DEFAULT_COEP`](#) value) is the gate — with COOP — for cross-origin ISOLATION (`crossOriginIsolated`, the prerequisite for `SharedArrayBuffer` / high-res timers). It is off because `require-corp` BREAKS any cross-origin subresource lacking its own opt-in (CORP/CORS) header. **Opt in:** `coep: true` (only after every cross-origin asset you load sends the opt-in header); `coep: '<directive>'` overrides the emitted value (e.g. `credentialless`).
- **HSTS (`hsts`)** — **OFF by default**. Strict-Transport-Security is meaningful ONLY over HTTPS and is browser-PINNED — a wrong value is STICKY (a client remembers it for the whole `max-age`), so shipping it on an HTTP-reachable host can lock users out. **Opt in:** `hsts: true` sets [`DEFAULT_HSTS`](#) = `max-age=31536000; includeSubDomains` (one year); `hsts: '<directive>'` overrides it (e.g. to add `; preload`). Turn it on only once your origin is HTTPS-only and you have confirmed every subdomain is too (`includeSubDomains` covers them all).
- **`Referrer-Policy` (`referrer`)** — default [`DEFAULT_REFERRER_POLICY`](#) = `strict-origin-when-cross-origin` (full URL same-origin, only the origin cross-origin, nothing when downgrading HTTPS→HTTP — the modern privacy default). **Loosen/change:** `referrer: 'no-referrer'` (send none) or any policy token; `referrer: false` omits it.
- **`Permissions-Policy` (`permissions`)** — default [`DEFAULT_PERMISSIONS_POLICY`](#) = `camera=(), microphone=(), geolocation=()` (deny those powerful features to the page). **Loosen:** `permissions: 'camera=(self), geolocation=()'` to grant a capability the app actually uses; `permissions: false` omits it (the browser default-allows per its own policy).
- **`X-Request-ID` (`requestId`)** — ON by default (the §4.2.1 entity-keyed group). It mints a fresh `crypto.randomUUID()`, sets it on the response ([`REQUEST_ID_HEADER`](#) = `x-request-id`), and stashes it on `context.state` under [`REQUEST_ID_STATE`](#) for downstream logging / correlation. By default an INCOMING `X-Request-ID` is IGNORED (anti-spoof). **Trust an upstream id:** `requestId: { trust: true }` echoes a client-supplied id — but ONLY one matching the strict charset [`isValidRequestId`](#) = `^[A-Za-z0-9_-]{1,200}$` (no CR/LF, control chars, whitespace, or over-length); an absent / malformed / oversized id is rejected and a fresh UUID minted, so a CRLF / log-injection / header-smuggling payload can NEVER ride into the response header or `context.state`. Only set `trust` behind a trusted proxy that mints (or sanitizes) the id, since `trust` honors a client-controlled value. `requestId: false` disables the id entirely.

### Cookies (`CookieOptions`)

Every cookie the spine writes — the signed session id, the CSRF token — goes through [`serializeCookie`](#) with secure-leaning attribute defaults.

- **`secure: 'auto'` (the default for the session / CSRF writers)** — TLS-aware: emits `Secure` when the request arrived over TLS ([`requestEncrypted`](#) → `request.socket.encrypted === true`), and OFF over plaintext HTTP — resolved per-request by [`resolveSecure`](#). So a direct-HTTPS deployment is `Secure` automatically while local-dev HTTP still works, with NO per-environment config. **Force it:** `secure: true` when a TLS-terminating proxy re-presents the request to node as plaintext HTTP (where `'auto'` would read no TLS and wrongly drop `Secure`); `secure: false` to force it off. NOTE: a `sameSite: 'None'` cookie is ALWAYS `Secure` regardless (the spec/browser requirement — [`serializeCookie`](#) adds it).
- **`HttpOnly` (default ON)** — hides the cookie from client JS (`document.cookie`), so an XSS payload can't read the session id. The CSRF token cookie deliberately sets `httpOnly: false` so the double-submit token IS readable by client JS to echo. Leave session cookies `HttpOnly`.
- **`SameSite` (default `Lax`)** — `Lax` sends the cookie on top-level navigations but not most cross-site subrequests (a baseline CSRF mitigation); the CSRF cookie uses `Strict`. **Change:** `sameSite: 'Strict'` (never sent cross-site — tightest, but breaks inbound links that expect the session) or `sameSite: 'None'` (sent cross-site — required for a cross-origin embed, and FORCES `Secure`).
- **`__Host-` / `__Secure-` cookie-name prefixes** — if you name a cookie with one of these browser-enforced prefixes, the inbound parse protects its integrity: [`parseCookies`](#) REJECTS a non-token or whitespace-padded NAME ([`isCookieName`](#)), so a smuggled `'  __Host-x'` can't be trimmed into the prefix-protected name. The `Domain` / `Path` attributes are also injection-validated ([`isCookieAttribute`](#) — a `;` / `,` / whitespace / control char throws rather than injecting another attribute).

### CSRF (`createCSRFGuard`)

`createCSRFGuard({ secret })` is the stateless **double-submit signed-cookie** defense — NO server-side store (the signed cookie IS the state). On a SAFE method ([`DEFAULT_CSRF_SAFE`](#) = `GET` / `HEAD` / `OPTIONS`) it mints a token, sets it as a signed cookie ([`DEFAULT_CSRF_COOKIE`](#) = `csrf`, `SameSite=Strict`, `HttpOnly:false`, `Secure` auto), and exposes the RAW token on `context.state` under [`CSRF_STATE`](#) = `csrf` for the handler to embed. On a MUTATING method it reads the submitted token (from [`DEFAULT_CSRF_HEADER`](#) = `x-csrf-token`, else the parsed-body [`DEFAULT_CSRF_FIELD`](#) = `_csrf`), verifies the signed cookie, and `403`s on a missing / invalid / mismatched token.

- **Session-binding (ON when a session precedes the guard) — the REQUIRED ordering.** When a session is on `context.state` under [`SESSION_STATE`](#) (the key `createSession` / `createCookieSession` set), the minted token is BOUND to the session id (`signToken(sessionId)`), so a mutating request must present the token bound to ITS OWN session — a cookie-injection attacker (sibling subdomain / MITM) who plants both double-submit halves under another user's session is STILL `403`ed. **Mount the session middleware BEFORE the CSRF guard** to activate this; the guard reads the session it pairs with from the state the session middleware already set.
- **Sessionless fallback (weaker — document it).** With NO session ahead of it, the guard FALLS BACK to a signed-random token (the plain double-submit). This is weaker — it has no per-user binding, so a write-cookie attacker who plants both halves is not caught. This is exactly why the session ordering is required for the strong guarantee; rely on the fallback only when there is genuinely no session layer.
- **How the client echoes the token.** On a safe `GET` the handler reads the raw token off `context.state.get(CSRF_STATE)` and embeds it (a hidden form field, a `<meta>` tag, a JSON bootstrap) — OR client JS reads the non-`HttpOnly` `csrf` cookie. The client then sends it back on every mutating request, either in the `x-csrf-token` header or the `_csrf` body field. Reading the body field requires the body already parsed, so **mount [`createBodyParser`](#) ahead** of the guard when the token rides a form field. Register [`createErrorBoundary`](#) OUTSIDE the guard so the `403` renders.

### Sessions (`createSession` / `createCookieSession`)

Sessions live until an explicit end by default (no TTL). Harden the lifetime and durability with three knobs (Contract clauses 20–23):

- **Idle `ttl` (ms).** `ttl: <ms>` evicts a session not accessed within the window — read as ABSENT and dropped on the next access (lazy, no background timer). Every live access TOUCHES it, so an active session never idle-ages-out. Set it so an abandoned session can't be resumed indefinitely.
- **`absoluteTtl` (ms from mint).** `absoluteTtl: <ms>` caps the TOTAL lifetime from mint, INDEPENDENT of the idle touch — a session older than `absoluteTtl` is evicted EVEN IF continuously active, so a stolen / hijacked id can't be kept alive forever just by being used (the idle `ttl` alone would). `createdAt` is stamped once at mint and preserved across re-persists, so activity never slides the absolute window.
- **`regenerate()` on privilege change / login (anti-fixation).** After authentication or any privilege escalation, call `session.regenerate()` (via the [`SessionControlInterface`](#) on `context.state`) to ROTATE the id — it mints a fresh id, carries the `data` across, and drops the old id, so a session-fixation attacker who planted a known id pre-login can't ride it post-login. `destroy()` ends the session (logout) and clears the cookie.
- **A durable `store` for restart-persistence.** The default store is in-memory ([`createMemorySessionStore`](#)) — sessions vanish on restart. Supply `store: createDatabaseSessionStore({ … })` (or any [`SessionStoreInterface`](#)) to persist across restarts (or share one store across middlewares) without touching `createSession` — the store OWNS the TTL sweep. Persist a PLAIN JSON-serializable session record (the default `Session` class's `#private` id does not round-trip through a JSON column).

### CORS (`createCors`)

`createCors()` defaults `origin: '*'` — a constant allow-all with NO reflection and NO credentials. **To allow specific origins**, pass `origin: ['https://app.example.com', …]`: the response then REFLECTS the request `Origin` when it is on the list (Contract 13). Two hardenings ride the allow-LIST path automatically: the response carries `Vary: Origin` (appended via [`mergeVary`](#) without duplication) so a shared cache can't serve one origin's allowed response to another (cache poisoning); and the literal `Origin: null` is NEVER reflected ([`resolveOrigin`](#) drops it unconditionally, even if `'null'` is on the list), so a sandboxed iframe / `file://` / opaque origin can't be granted access. A `'*'` / single-origin policy is a constant and adds no `Vary`. If you reflect specific origins to support credentialed requests, keep the allow-list TIGHT — every listed origin is fully trusted with your responses.

### Rate limiting (`createRateLimiter`)

`createRateLimiter({ max, window })` keys the per-client window on the **socket peer IP**, NOT a forwardable header. The default key ([`resolveClientKey`](#)) is the verified token (when an upstream [`createTokenGuard`](#) stashed one under [`TOKEN_STATE`](#)) else the raw socket peer IP — and `X-Forwarded-For` is deliberately NOT consulted because it is client-SPOOFABLE: an attacker who could set it would mint unlimited identities and dodge the limit. The IP is collapsed to its bucket via [`clientRateKey`](#) — an **IPv6 address to its `/64` network** (so a user can't rotate the host bits of their `/64` allocation to evade the cap), an IPv4 / IPv4-mapped address unchanged.

- **Behind a TRUSTED proxy.** When you terminate at a proxy/load-balancer that sets a verified client header, supply your own `key: (context) => …` to read it — e.g. `key: (context) => flattenHeaders(context.request.headers)['x-real-ip'] ?? ''`. **Caution:** trusting ANY forwarded header is only safe if a trusted proxy SETS it and strips a client-supplied forgery; without that, a custom `key` over a forwardable header re-opens the spoofing hole the default avoids. Bound memory with `capacity` (default [`DEFAULT_RATE_LIMIT_CAPACITY`](#) = `10_000`).

### Body & decompression (`createBodyParser`)

`createBodyParser({ limit, decompression })` bounds two sizes (Contract 27):

- **`limit` (the compressed/wire body cap).** Default [`DEFAULT_BODY_LIMIT`](#) = `1_048_576` (1 MiB). A request body over this is a `413` ([`ContentTooLargeError`](#)). Raise it (`limit: <bytes>`) for an endpoint that accepts large payloads; keep it as tight as the endpoint truly needs.
- **`decompression` (the zip-bomb cap).** A request declaring `Content-Encoding: gzip`/`deflate`/`br` can be tiny on the wire yet inflate enormously. The decompressed size is capped by feeding `node:zlib`'s `maxOutputLength`, which ABORTS the inflation the instant the output would exceed the cap — it never materializes the full bomb ([`decompressBody`](#)). For a MOUNTED parser, `decompression` DEFAULTS to its own `limit` ("limit means limit"), so a tight `limit` bounds the decompressed body too. **Loosen:** set `decompression: <bytes>` independently — and larger — to allow a body that legitimately inflates above the compressed-byte `limit`. A NON-POSITIVE value `decompression: 0` means UNCAPPED decompressed output (no `maxOutputLength` fed to zlib) — use ONLY when the compressed input is ALREADY bounded by `limit`, since uncapped inflation forfeits the zip-bomb guard. (The bare `context.body()` path, with no parser mounted, falls back to [`DEFAULT_DECOMPRESSED_LIMIT`](#) = `16_777_216` = 16 MiB.)

### Static files (`createStatic`)

`createStatic({ root })` serves a directory over `node:fs` with three security-critical guards that have NO opt-out (Contract 24):

- **Traversal guard.** The request path is stripped of `prefix`, `decodeURIComponent`d (a malformed escape ⇒ not-found), NUL-byte-refused, made relative, normalized, then resolved UNDER the already-resolved `root` and REQUIRED to equal `root` or sit beneath `root + sep` ([`resolveStaticPath`](#)). So a `../`, an encoded `..` (`%2e%2e`), an absolute decoded path, a NUL byte, or a sibling that merely shares the root's string prefix all resolve OUTSIDE and are refused — the file is never `stat`ed, never read. It is TOTAL (adversarial input never throws).
- **Windows reserved-device-name rejection (CVE-2025-27210).** A served path segment matching a legacy DOS device name ([`RESERVED_DEVICE_NAMES`](#) = `CON` / `PRN` / `AUX` / `NUL` / `COM1`–`COM9` / `LPT1`–`LPT9`, plus `CON.txt` / `AUX.` and the superscript `COM¹` variants, case-insensitive — [`isReservedDeviceName`](#)) is REFUSED, so a device alias can't side-step the traversal containment to reach a hardware device instead of a file.
- **SPA-fallback segment boundary.** When `fallback` is enabled, the `/api` exclude (and any custom `exclude`) is matched on a SEGMENT boundary ([`isUnderPath`](#)) — `/apifoo` is NOT under `/api`, so a near-miss path can't slip past the exclude and wrongly receive the SPA shell instead of a `404`.
- **Single-range only.** A multi-range `Range` request is served as the FULL `200`, never a multi-part body — the range parser ([`parseRange`](#)) handles only a single satisfiable / unsatisfiable range and is TOTAL (a reversed / over-large / malformed range yields the full response, never a throw), avoiding the multi-range amplification surface.

The `dotfiles` policy (default `'ignore'`) and `fallback` tune behavior, not the guards: `dotfiles: 'deny'` `403`s a `.env` / `.git` request, `dotfiles: 'allow'` serves it (do this only deliberately).

### Uploads (`createMultipart`)

`createMultipart({ allowed, limits, directory })` stream-parses `multipart/form-data` with the bytes — never the client's word — as the source of truth (Contract 29):

- **Sniff-authoritative `allowed` list — the declared `Content-Type` is NOT trusted.** With `allowed: ['image/png', …]`, each file's REAL type is sniffed from its leading magic bytes ([`detectMIME`](#) over [`MAGIC_BYTES`](#)) and must BOTH be on the list AND agree with the declared `Content-Type`. So a polyglot / type-confusion upload — `<script>` bytes declared `image/png` — is REJECTED (`415`, its temp file cleaned), because the decision rests on the bytes, never the client-controlled declared type. Without `allowed`, every file is staged and its `mime` + `validated` recorded, but none is rejected on type. An EMPTY `allowed: []` allows NOTHING.
- **The signature-less-type limitation.** [`MAGIC_BYTES`](#) covers the common binary upload types (JPEG / PNG / GIF / WebP / PDF / ZIP). A type with NO magic signature — plain text, CSV, SVG, most `application/*` text formats — cannot be sniffed, so it can't be put on the `allowed` list and have it pass (an un-sniffable type is recorded `validated: false` and accepted ONLY when no whitelist applies). To accept a signature-less type, either run WITHOUT a whitelist (accepting that the type is unverified) or validate it in your handler after staging.
- **Size limits (DoS bound, enforced MID-STREAM).** [`MultipartLimits`](#) — `file` (default 10 MiB), `files` (10), `field` (64 KiB), `fields` (100), `total` (52 MiB) — each trips the moment it is exceeded: reading STOPS, every already-staged temp file is DELETED, and a `413` is thrown, so a hostile body is never buffered past the bound.
- **Random temp-name staging.** Each file streams to a temp file under a RANDOM name (`crypto.randomUUID()` in `directory`, default `os.tmpdir()`). The untrusted client `filename` is METADATA only (the `name` field) and is NEVER used to build the path, so a `filename="../../etc/passwd"` cannot escape the staging dir (the path-traversal defense). Cleanup leaks no temp file on a limit breach, parse error, OR client disconnect mid-upload.

### Body prototype-pollution scrub

A parsed JSON body is recursively stripped of the prototype-pollution keys `__proto__` / `constructor` / `prototype` in place ([`scrubPrototype`](#) over [`isDangerousKey`](#)), applied by [`decodeBody`](#) on the one cached `body()` path — so a `{"__proto__":{"polluted":1}}` body (or a urlencoded `__proto__[x]=1`, or a multipart `__proto__` field) never reaches `Object.prototype` nor rides into the parsed value. This is unconditional, with no opt-out.

### Error boundary (`createErrorBoundary`)

`expose: false` (the default) renders a non-`HTTPError` throw as a GENERIC `500` body, keeping the real error message + stack server-side — so an internal fault (a DB string, a file path, a stack frame) is never leaked to the client. An `HTTPError`'s OWN message is always sent (it is the handler's intentional client-facing signal). **Loosen for dev only:** `expose: true` echoes a non-`HTTPError`'s message in the `500` body — useful locally, but never in production (it leaks internal detail). Pair it with `report` (a fire-and-forget sink) to log the full error regardless.

### Loosening responsibly

These defaults are secure ON PURPOSE — each is the strict choice because the alternative carries a real, named exposure. When you override one for your app:

- **Understand the exposure you are accepting.** Each subsection above states the trade-off — `csp: false` forfeits injection hardening, `corp: 'cross-origin'` lets any origin embed the response, `decompression: 0` removes the zip-bomb guard, a custom rate-limit `key` over a forwardable header re-opens spoofing. Loosen only when you understand and accept that specific exposure.
- **Scope it as narrowly as possible.** Prefer a per-route relaxation (a dedicated `createStatic` / `createSecurity` instance on a route group) over loosening the whole server; prefer extending a policy (a tighter `csp` that ADDS one CDN source) over disabling it; prefer `coop: 'same-origin-allow-popups'` over `coop: false`. The narrowest override that unblocks your case leaves the rest of the secure default intact.
- **Re-tighten when the constraint lifts.** A `secure: false` for local HTTP, an `expose: true` for debugging, a `corp: 'cross-origin'` for a migration — each should revert the moment its reason is gone.
